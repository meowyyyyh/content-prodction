# 图文排版智能化方案 — 三引擎协同

> 2026-07-16 迭代 V2 · 三引擎各司其职，排版从规则驱动进化为数据驱动

---

## 一、现状与问题

当前排版靠硬编码：

```js
// generator.js 里的 imageModuleMap
const imageModuleMap = {
  '封面图': ['hook'], '产品图': ['taste'],
  '配料表': ['ingredient', 'trust', 'origin'],
  '场景图': ['scene'], '品牌图': ['brand'], '包装图': ['hook'],
}
// prompt 里的排版指令
'请严格按 图1→文→图2→文→... 的节奏写对应文案'
```

三个致命缺陷：

1. **看不见图片内容** — "产品图"三个字无法区分"正面特写"和"倒出瞬间"，它们需要的图文配合完全不同
2. **语料再多也不会变好** — 规则写死的，不会从数据里学习排版模式
3. **排版和文案脱节** — 固定节奏不考虑每段文案实际在描述什么，读起来生硬

---

## 二、三引擎各司其职

| 引擎 | 擅长 | 当前只做 | 应该做 |
|------|------|------|------|
| doubao（视觉） | 看见真实像素，理解构图和画面关系 | 类型分类+一句话描述 | **每张图的布局角色判断**（hero/detail/scene/info/step） |
| DeepSeek（文本） | 理解上下文，知道自己在写什么 | 按固定模板生成文案 | **写作的同时，用 inline markers 自然完成排版** |
| 语料库（统计） | 积累历史经验，告诉系统"这个品类通常怎么排" | 文本写作因子 | **同时输出布局因子**（图片数、分组偏好、图文流） |

**核心原则：doubao 做图片级判断，DeepSeek 做段落级编排，语料库做品类级校准。**

---

## 三、架构图

```
图片上传
   │
   ▼
doubao（每张图独立）──→ { type, desc, layout_role }
                           layout_role: hero | detail | scene | info | step
                           （一次调用，几乎零边际成本）
   │
   ▼
Prompt 注入层 ──→ { layout_role + 语料布局因子 + 写作因子 + 语料few-shot }
                    │
                    ▼
DeepSeek ──→ 文案 + 排版标记 [IMG:1 cover] [IMG:2 IMG:3 pair]
              │  在写"开盖香气"时，自然决定图1放在这里；在写"质地细节"时，自然决定图2-3放在那里
              │
              ▼
parseModuleResults → 结构化分段
  { moduleKey, segments: [{ type: 'text' | 'image_slot', content, imgIds, layout }] }
              │
              ▼
CenterPanel 渲染 → cover 全宽大图 / pair 双图并排 / grid 多图网格 / inline 图文混排
              │
              ▼
运营编辑 → 导出 → 自动写入语料库（带完整布局元数据）
              │
              ▼
语料越来越丰富 → 布局因子越来越准 → 排版越来越自然 🌀
```

## 四、四阶段执行计划

### Phase 0：零语料启动（2-3 天）

**目标：不依赖任何语料，立即让排版从"所有图堆在一起"变成"有节奏的图文编排"。**

#### 改动 1：doubao 分类 prompt 扩展

当前 vision.js 输出 `{"type":"产品图","desc":"白色酸奶瓶"}`，加 `layout_role`：

```
请分析这张图片：
1. 类型：产品图/配料表/场景图/品牌图/包装图/其他
2. 一句话描述
3. 布局角色（layout_role）：
   hero   — 主视觉，占全宽大空间（产品正面、包装展开）
   detail — 细节特写，适合和同类分组排列（微距、质地、纹理）
   scene  — 生活场景，独立段落（办公桌、餐桌、户外）
   info   — 信息图/配料表/文字说明，嵌入段落（截图、表格、标识）
   step   — 步骤图/流程，适合横向连排（开箱、冲泡、使用过程）

返回 JSON：{"type":"产品图","desc":"白色酸奶瓶正面，绿色标签","layout_role":"hero"}
```

**成本变化**：max_tokens 从 60 提到 80，几乎不增加费用。不改变调用逻辑，每张图仍然是一次独立调用。

#### 改动 2：取消硬编码排版指令

删除 generator.js 里的 "请严格按 图1→文→图2→文→... 的节奏写对应文案"。

改为注入 layout_role 信息，让 DeepSeek 根据角色自然编排：

```
taste 模块共有 8 张图片，每张有不同布局角色：
- 图1（hero）：主视觉
- 图2-5（detail × 4）：细节特写
- 图6-7（scene × 2）：使用场景
- 图8（info）：配料表

每段文案要自然地对应它所描述的图片。不要固定"图→文→图→文"节奏，
让每张图出现在它内容上最相关的那段文字旁边。
```

#### 改动 3：前端 layout_role 差异化渲染

CenterPanel 接收每张图的 layout_role：

| layout_role | 前端渲染 |
|------|------|
| hero | 全宽大图，圆角，限高 400px，类似杂志封面 |
| detail | 2-4 张网格排列（CSS grid），点击可放大 |
| scene | 独立图文块，图在文前或文后，有间距 |
| info | 小图嵌入文字流，max-width 200px，float |
| step | 横向连排，带步骤编号 |

#### Phase 0 交付效果

同一批 8 张产品图，不再机械穿插，而是：

> [hero 全宽封面] → 开篇文案 → [detail 四图网格] → 口感描述 → [scene 双图并列] → 使用场景 → [info 配料表嵌入] → 信任段落

### Phase 1：语料布局因子提取（4-5 天）

**目标：语料积累后，排版不再只看单张图片的角色，还能感知"这个品类下 taste 模块通常怎么排"。**

#### 改动 1：语料 schema 扩展

```ts
interface CorpusEntry {
  category: string
  module_id: string
  content: string
  layout_meta?: {
    image_count: number
    image_grouping: 'single' | 'pair' | 'grid' | 'carousel' | 'contiguous'
    image_text_flow: 'before' | 'after' | 'mixed'
    density: 'high' | 'medium' | 'low'
    image_roles: string[]      // 每张图的 layout_role
    module_order: number
  }
}
```

首批标注：当前仅有 1 篇语料，手动标 5 分钟。之后每次运营导出的笔记自动带 layout_meta。

#### 改动 2：analyzeLayoutFactors

仿照现有 `analyzeCorpusFactors`（写作因子），新增布局因子分析：

```
语料排版因子分析

### taste 模块 · 排版因子（N篇参考）
| 参数 | 值 | 说明 |
|------|-----|------|
| 平均图片数 | 5.6张 | 配图数量范围 |
| 排列模式 | 连续(83%) | 连续 vs 分散 |
| 图文位置 | 图在文前(75%) | 前置 vs 后置 vs 混合 |
| 分组偏好 | pair(45%) > grid(30%) | 图片如何分组 |
| 密度 | 高 | 图占模块面积比例 |

### hook 模块 · 排版因子...
```

#### 改动 3：注入 prompt

布局因子和写作因子一起注入 prompt，供 DeepSeek 参考：

```
以下是你需要遵循的品类排版规律（基于 X 篇语料统计）：
  taste 模块：平均 5-6 张图，83% 连续排列，75% 图在文前，pair 分组最常见
  hook 模块：平均 1-2 张图，90% 单张全宽
  scene 模块：平均 2-3 张图，独立段落排列
```

### Phase 2：DeepSeek inline markers 结构化排版（5-7 天）

**目标：排版从"隐式自然编排"升级为"显式结构化输出，前端精确渲染"。**

#### 改动 1：prompt 注入排版标记格式

告诉 DeepSeek 在文案中插入标记来控制图片位置：

```
请在文案中插入排版标记来控制图片在最终页面中的位置。

标记格式：
  [IMG:<id> <layout>]   — 单张图
    layout: cover | pair | inline | full
    示例：[IMG:1 cover] 图1全宽大图

  [IMG:<id> <id> <layout>]  — 多张图
    layout: pair | grid
    示例：[IMG:2 IMG:3 pair] 图2图3并列

每张图应该出现在文案中描述它的那段文字旁边。
不一定要每段都配图。图片数量决定了标记数量，请全部覆盖。

语料布局因子：
  taste 模块平均 5.6 张图，pair 分组最常见

本次 taste 模块图片布局角色：
  图1（hero）图2-5（detail×4）图6-7（scene×2）图8（info）
```

DeepSeek 输出示例：

```
===taste===
[IMG:1 cover]
开盖就能闻到一股清新的木姜子香气...
[IMG:2 IMG:3 pair]
倒进杯子里质地很浓稠，拉丝感满满...
[IMG:4 IMG:5 IMG:6 IMG:7 grid]
四款口味各有特色，从左到右依次是...
[IMG:8 inline]
配料表非常干净，生牛乳排第一...
```

#### 改动 2：parseModuleResults 升级

现有 parser 只做 `===moduleKey===` 分割。升级为：

```ts
interface StructuredSegment {
  type: 'text' | 'image_slot'
  content?: string       // 文本内容
  imgIds?: number[]      // 对应图片 ID
  layout?: 'cover' | 'pair' | 'grid' | 'inline' | 'full'
}

interface StructuredModule {
  moduleKey: string
  segments: StructuredSegment[]
}
```

解析逻辑：
1. 按 `===moduleKey===` 分割模块
2. 按 `[IMG:...]` 标记分割段落
3. 标记前的文本为 text segment，标记本身转化为 image_slot segment
4. LLM 未覆盖的图片（遗漏）：按 layout_role 兜底，默认 detail 图以 grid 排在模块末尾

#### 改动 3：CenterPanel 渲染升级

根据 segments 数组渲染图文混排：

| layout | 渲染方式 |
|------|------|
| cover | 全宽大图，高度自适应，max-height: 400px |
| pair | flex row，两张图各占 50%，gap 8px |
| grid | CSS grid，imgIds.length 决定列数（4张→2列，6张→3列） |
| inline | 图片嵌入文字流，max-width: 200px，float left |
| full | 独占一行，全宽 |

渲染时保持**拖拽可调**——标记只是"AI 建议"，运营可以拖动图片到不同模块或调整顺序。

#### 改动 4：SSE 流式兼容

当前生成结果是流式推送的（`/api/generate/stream`）。markers 和文案一起流式到达。CenterPanel 的 `typewrite` 效果需要同时渲染文本和图片占位——图片先显示 skeleton，等 markers 确认后渲染真实图片。

### Phase 3：数据飞轮闭环（持续，可与 Phase 2 重叠）

#### 改动 1：导出自动入库

每次运营导出笔记时，同时写入 `data/corpus/`：

```json
{
  "category": "调制乳/风味牛奶",
  "module_id": "taste",
  "content": "[IMG:1 cover]\n开盖就能闻到...\n[IMG:2 IMG:3 pair]\n倒进杯子...",
  "layout_meta": {
    "image_count": 8,
    "image_grouping": "mixed",
    "image_text_flow": "interleaved",
    "density": "high",
    "image_roles": ["hero","detail","detail","detail","detail","scene","scene","info"],
    "module_order": 2
  }
}
```

#### 改动 2：排版采纳信号采集

| 运营行为 | 信号含义 |
|------|------|
| 直出+导出，未调整排版 | 正信号：此排版方案好 |
| 拖拽调整了某张图位置 | 负信号：原布局不理想 |
| 在某模块删了图 | 负信号：此模块图片过多 |
| 在某模块手动插入了图 | 负信号：此模块缺图 |

这些信号累计到 `localStorage['flywheel_scores']`，下次生成时影响布局因子权重。

#### 改动 3：飞轮演化示意

```
第 1 次生成 → 只有 layout_role 驱动 → 排版偏直觉
第 10 次生成 → 有了 9 篇语料布局因子 → 排版开始对齐品类习惯
第 50 次生成 → 50+ 篇语料 + 运营采纳信号 → 排版越来越贴近该运营的真实偏好
第 200 次生成 → 品类、风格、个人三轴布局画像 → 排版"猜你喜欢的水平"
```

## 五、与原始方案（V1）的差异

| 维度 | V1（doubao 全权排版） | V2（三引擎协同） |
|------|------|------|
| 排版权 | doubao 输出完整 JSON | DeepSeek 在写作中用 inline markers 自然生成 |
| doubao 调用 | 每张图一次 + 二次批量调用 | 仅每张图一次，加一个 layout_role 字段 |
| 图文一致性 | doubao 不知文案内容，可能脱节 | 同一模型决定，天然对齐 |
| 技术风险 | 轻量模型做复杂空间推理，效果不确定 | 每个模型做自己擅长的事，风险可控 |
| 语料利用 | 语料统计只做 doubao 的参考输入 | 语料布局因子直接注入 DeepSeek，和写作因子同层 |
| 前端改动 | 解析 doubao JSON → 渲染 | 解析 DeepSeek markers → 渲染，更贴近现有 parse 逻辑 |

## 六、与当前实现的关系

| 当前 | Phase 0 后 | Phase 1 后 | Phase 2 后 | Phase 3 后 |
|------|------|------|------|------|
| 硬编码排版规则 | layout_role 驱动自然编排 | + 语料布局因子注入 | + inline markers 结构化输出 | + 数据飞轮闭环 |
| 所有图机械穿插 | hero/detail/scene 差异化 | + 品类历史规律 | + 精确排版+渲染 | + 个人偏好进化 |
| 不会从数据中学习 | — | 开始感知品类习惯 | 精确控制每张图位置 | 越用越像运营的习惯 |
