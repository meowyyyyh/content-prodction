## 一、产品目标

将"快稿种草小助手"从 **AI 文案助手（1.0）** 升级为 **图文智能工作台（2.0）**：

- 运营上传商品图片 → AI 自动识别图片类型并生成描述 → 文案与图片穿插排版 → 所见即所得编辑 → 导出
- MVP 范围：调制乳/风味牛奶一个品类簇
- 同步启动数据飞轮 Phase 0（信号采集基础设施，只写不读）

## 二、三版定位与默认风格

### 2.1 三版架构

```
┌──────────────┬──────────────┬──────────────┐
│    V1 固定    │   V2 可换    │   V3 可换    │
│   默认风格    │  Elo 第 1    │  Elo 第 2    │
├──────────────┼──────────────┼──────────────┤
│  初始：小红书  │  初始：闺蜜风  │  初始：趣味风  │
│  随时间进化   │  Elo 动态排序  │  Elo 动态排序  │
│  越来越像"你" │              │              │
└──────────────┴──────────────┴──────────────┘
```

| 版本 | 位置 | 内容 | 变化机制 |
|------|:--:|------|------|
| V1 默认风格 | 固定第 1 | 初始小红书，随时间进化为个人复合风格 | 模块偏好追踪 + L2 画像注入 |
| V2 | Elo 第 1 | 纯风格（闺蜜风/高端风/…） | Elo 排名动态排序 |
| V3 | Elo 第 2 | 纯风格 | Elo 排名动态排序 |

### 2.2 默认风格进化机制

默认风格是第七种风格——不是写死的 prompt 模板，而是**每次生成前根据用户行为数据实时拼装的动态 prompt**。

```
运营从 V2（闺蜜风）采纳了 taste 模块 → 重复多次
  → 默认风格的 taste 模块来源 = "girlfriend"
  → 下次生成 V1 时，taste 用闺蜜风的规则写

运营在中栏总是把 hook 的 emoji 删掉、把 "姐妹们" 改掉
  → L2 画像提取：emoji 低、不用 "姐妹们"
  → 注入 V1 prompt，叠加到所有模块
```

**数据结构**：

```js
// localStorage['flywheel_scores:{userId}'] 新增
"default_modules": {
  "食品保健/咖啡麦片冲饮/常温乳制品/调制乳-风味牛奶": {
    "taste":  { source: "girlfriend",  n: 12 },
    "price":  { source: "premium",     n: 8  },
    "hook":   { source: "xiaohongshu", n: 15 },
    "trust":  { source: null,          n: 0  },  // 无偏好，回退基础风格
  }
}
```

**冷启动与退化**：

| 场景 | 行为 |
|------|------|
| 新用户 / 该模块从未被采纳 | 使用初始基础风格（小红书种草风） |
| 采纳 ≥ 5 次且 ≥ 80% 来自同一风格 | 完全使用该风格的规则 |
| 采纳 3-4 次，来源分散 | 使用偏好风格 + 语气向基础风格靠拢 20% |
| 采纳 < 3 次 | 使用基础风格 |

### 2.3 V2/V3 Elo 排序

V1 不参与 Elo 竞争，V2/V3 由 Elo 排名驱动：

| 事件 | Elo 变化 | 说明 |
|------|------|------|
| 采纳版本（V2/V3） | 该版本战胜另一个版本 | V2/V3 竞争 |
| 点踩版本（V2/V3） | 该版本输给中性基准 | 明确否定 |
| 快捷按钮改写 + 导出 | 目标风格权重 ×2 | 运营主动变换，意愿更强 |
| 导出 | 终稿风格额外加分 | 最强正信号 |

## 三、视觉模型接入方案

### 3.1 模型选型

| 项目 | 选型 |
|------|------|
| 模型 | **doubao-seed-2.0-lite**（火山方舟） |
| 用途 | 图片分类 + 一句简短描述 |
| 输入 | 0.6 元/百万 token |
| 输出 | 3.66 元/百万 token |
| 单张图成本 | ≈ ¥0.00035（512 token 输入 + ~15 token 输出） |
| 批量推理折扣 | 5 折 |
| 上下文窗口 | 64K |

### 3.2 为什么需要视觉模型

| 能力 | 工具 | 产出 |
|------|------|------|
| 图片类型识别 | doubao-seed-2.0-lite | "产品图"、"配料表"、"场景图"、"品牌图" |
| 图片描述 | doubao-seed-2.0-lite（同一请求） | "白色酸奶瓶，绿色标签，200g" |
| 图片文字提取 | Tesseract.js（浏览器端） | "配料表：生牛乳、白砂糖…" |

分类 + 描述是一个 API 请求同时返回的，不额外调 API。

### 3.3 调用流程

```
运营上传图片
  → 浏览器端压缩至 512px（Web Worker）
  → 发送 base64 到 POST /api/images/classify
  → 后端批量调 doubao-seed-2.0-lite（10 张/batch，3 路并行）
  → 返回：[{ id, type: "产品图", desc: "白色酸奶瓶，绿色标签" }, ...]
  → 前端展示缩略图 + 标签 + 描述，运营可手动调整
```

### 3.4 OCR（浏览器端，不调 API）

```js
import Tesseract from 'tesseract.js'
const { data: { text } } = await Tesseract.recognize(imageFile, 'chi_sim+eng')
```

OCR 提取的文字（配料表、包装上的净含量等）注入 prompt，确保文案与图片信息一致。

### 3.5 成本估算

```
单张图：¥0.00035
90 张图：¥0.03
每天 5 篇笔记 × 90 张图：¥0.15/天
一个月 150 篇：¥4.5/月
批量 5 折后：¥2.25/月
一年：¥27/年
```

## 四、图文排版引擎

### 4.1 核心原则

1. **所有上传的图片全部展示**，不丢图、不折叠
2. **文对图写**：prompt 告诉 AI 每张图是什么，AI 对着每张图写对应文案
3. **图→文→图→文 穿插**：不是图归图、文归文两坨分开

### 4.2 prompt 拼装（新增图片描述注入）

```
现有 buildPrompt：
  systemParts → 风格规则 → 模块要求 → 输出格式

新增：
  如果有图片数据 → 每个模块注入对应的图片描述清单：
  
  "taste 模块共有 8 张产品图。请按 图1→文→图2→文→...→图8→文 的节奏写。
   每段文字 1-2 句话，描述你看到的图片细节。
   图 1：产品正面特写，白色酸奶瓶，绿色标签
   图 2：倒出酸奶瞬间，拉丝效果
   图 3：勺子舀起，浓稠质地
   ..."
```

### 4.3 模板引擎

```
品类 → 模板匹配表（研发定义）→ 每个模块的图文穿插规则

示例：调制乳模板
  hook    → 封面图（轮播，最多 3 张，全宽）
  taste   → 左图右文 / 右图左文交替
  trust   → 图标列表（配料表截图放文案下方）
  brand   → 全宽图片
  scene   → 右图左文
  ...
```

模板是纯规则引擎，不调 AI。品类→模板匹配表初期由研发手动定义，后期由语料统计驱动。

### 4.4 视觉模型与文本模型的协作

两个模型不直接对话。它们通过**图片元信息**这个中间层解耦：

```
┌─────────────────┐
│  doubao-seed     │  看图 → 分类 + 描述
│  2.0-lite        │  输入：压缩图片 base64
│  (视觉模型)       │  输出：{ type: "产品图", desc: "白色酸奶瓶正面" }
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  图片元信息       │  结构化 JSON，不依赖任何模型
│  (中间层)        │  [{ id, type, desc, ocrText, suggestedModule }]
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌───────┐ ┌──────────┐
│DeepSeek│ │模板引擎    │
│(文本)  │ │(纯规则)   │
│        │ │          │
│读元信息│ │读元信息    │
│→ prompt│ │→ 品类匹配  │
│→ 对图写│ │→ 图片定位  │
│→ 生成  │ │→ 穿插排布  │
│  文案  │ │           │
└───┬───┘ └─────┬─────┘
    │           │
    └─────┬─────┘
          ▼
    图文混排输出
```

| 角色 | 谁干 | 输入 | 输出 |
|------|:--:|------|------|
| 看图分类 | 视觉模型 | 压缩图片 | 类型 + 一句话描述 |
| 看图识字 | Tesseract.js（浏览器） | 原图 | 配料表文字、包装信息 |
| 汇入 prompt | 后端 buildPrompt | 图片元信息 + 商品信息 | 含图描述的完整 prompt |
| 写文案 | DeepSeek | 含图描述的 prompt | 14 模块流式文案 |
| 图文排布 | 模板引擎（浏览器） | 文案 + 图片元信息 | 图文穿插 HTML |

**视觉模型不知道 DeepSeek 的存在，DeepSeek 不知道图片长什么样。** 元信息是它们之间的唯一桥梁。这个设计的优势：换模型只换一边——换视觉模型不影响文本生成，换文本模型不影响图片处理。

### 4.5 完整流水线

```
运营填商品信息 + 上传图片 → 点「一键生成」

  [并行开始]
  ├─ 浏览器压缩图片（512px，瞬间）
  ├─ doubao-seed-2.0-lite 分类+描述（10张/batch × N batch，3 路并行）→ ~4s
  └─ Tesseract OCR（4 Worker 并行）→ ~12s
  [并行结束]

  汇入 buildPrompt：
    - 图片描述清单（新增）
    - 风格规则（不变）
    - 语料库 RAG（不变）
  
  DeepSeek 流式生成文本 → 10-30s
  
  模板引擎渲染图文混排 → < 0.1s

总耗时 ≈ OCR(12s) + 生成(10-30s) ≈ 25-45s
```

## 附录：关键术语速查

| 术语 | 含义 |
|------|------|
| 默认风格 | V1 固定位置，初始小红书，随时间进化为个人复合风格 |
| 纯风格 | 六种写死的 prompt 模板（小红书/闺蜜/趣味/简约/高端/团长） |
