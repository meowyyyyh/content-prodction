# 快稿种草小助手 — 深度规划

## 一、这个智能体最终会长成什么样子？

### 3 个阶段，3 个产品形态

```
第一阶段                  第二阶段                  第三阶段
┌─────────────┐    ┌─────────────────┐    ┌───────────────────┐
│  AI 文案助手  │ → │  图文智能工作台   │ → │  用户私人内容工厂   │
│             │    │                 │    │                   │
│  帮你写文案   │    │  帮你写 + 配图    │    │  全链自动化         │
└─────────────┘    └─────────────────┘    └───────────────────┘
```

### 1.0 AI 文案助手（已完成 ✅）

> **定位**：用户的 AI 写手。输入商品信息，输出完整笔记。

核心能力：文本生成、快捷指令、双版本对比、云端部署。

### 2.0 图文智能工作台（当前阶段）

> **定位**：从"写手"升级为"排版师"。输入商品信息 + 图片，输出图文混排笔记。

**MVP 定义**：常温乳制品 + 图片上传 → 视觉模型分类 → 规则插入 → 生成图文混排笔记。

| 能力 | 说明 | MVP 状态 |
|------|------|:--:|
| 品类 | 常温乳制品（每日吨吨案例类目） | 🎯 |
| 文本生成 | 填表单 → 双版本 → 采纳 → 编辑 | ✅ |
| 图片分析 | 上传 → 视觉模型分类 → 规则插入 | 🔜 |
| 语料检索 | 乳品 5 篇 → RAG 检索 | 🔜 |
| 埋点 | Umami 自部署 | 🔜 |
| 快捷指令 | 风格改写、加/去 emoji、扩充文案 | ✅ |
| 部署 | Cloudflare Pages + Render | ✅ |

**验证清单**：

| 验证点 | 怎么看 |
|------|------|
| 图片分类准吗？ | 运营上传图片 → 看自动归到哪个模块 |
| 图片位置对吗？ | 生成后预览 → 图片在模块里位置合理吗 |
| 语料有用吗？ | 有语料 vs 无语料，生成质量有提升吗 |
| 埋点有数据吗？ | Umami Dashboard 能看到事件吗 |
| 运营改得多吗？ | 编辑率 < 30% 算及格 |

**MVP 通过后两条线并行**：

```
MVP 验证通过 ✅
    │
    ├─→ 运营线：准备其余 13 簇的语料（按模板填，他们的任务）
    │
    └─→ 研发线：Agent 能力开发（你的任务）
         ├── 用户画像/记忆学习
         ├── 自检自修
         └── 编排器（品类自动识别 + 风格推断）
```

### 3.0 用户私人内容工厂

> **定位**：不是工具，是用户的一部分。从商品到发布，全链自动化。

新增能力：
- 多平台分发 → 同一篇笔记适配小红书/快团团/朋友圈格式
- 商品智能录入 → 拍包装袋、扫码、粘贴链接，自动提取所有字段
- 数据闭环 → 埋点数据驱动持续调优
- 用户知识库 → 所有历史笔记、产品偏好、客户画像

### 终极形态

> **用户把商品扔进去，出来就是可以直接发的笔记。用户只需要做两件事：确认和发布。**

## 二、技术方案选择

### 决策 1：视觉模型 — 免费够用 vs 付费高精？

| 维度 | 智谱 GLM-4V-Flash | 豆包视觉 | 通义千问 Qwen-VL-Plus |
|------|:--:|:--:|:--:|
| **价格** | **永久免费** | 免费额度 | ¥0.75/百万 Token |
| **单图响应** | ~1.8s | **~0.8s** | ~2.7s |
| **准确率** | 86% | 高 | **92%** |
| **并发** | 10 | 5 | 不限 |
| **多图输入** | 单图 | 单图 | 多图 |
| **接入难度** | 换 URL+Key | 换 URL+Key | 换 URL+Key |
| **月费(POC)** | **¥0** | ¥0 | ~¥5-10 |

**待定**：会上根据运营对图片处理的实际需求再评估选哪个。POC 阶段可以先用 GLM-4V-Flash 免费测试。

### 决策 2：图片插入位置 — AI 决定 vs 规则引擎？

| 方案 | 说明 | 优势 | 劣势 |
|------|------|------|------|
| AI 决定 | 图片给模型，让它决定插入点 | 更智能 | 不可控、贵 |
| **规则引擎** | 标签→模块→顺序位置 | 可控、可预测、免费 | 不够灵活 |
| 混合 | AI 建议 + 规则兜底 + 运营拖拽 | 最优 | 实现稍复杂 |

**推荐**：规则引擎。运营手动拖拽更方便。

### 决策 3：商品录入 — 怎么更快？

| 方案 | 说明 | 状态 |
|------|------|:--:|
| A. 手动填写 | 现状 | ✅ |
| B. 文件解析 | 上传 Word/Excel | ✅ |
| C. 粘贴链接 | 粘贴商品链接，自动抓取 | 🔜 |
| D. OCR 识别 | 拍包装袋识别配料表 | 后期 |
| E. 平台 API | 调拼多多开放平台 | 长期 |

### 决策 4：类目体系 — 做到多细？

拼多多食品类目到三级有几百个，全做不现实。采用**品类簇**方案：按"写法像不像"分组，13 个簇覆盖所有食品品类。

见第三部分。

### 决策 5：评测体系

| 方案 | 说明 | 成本 |
|------|------|:--:|
| 运营主观评分 | 每次生成后打分 1-5 | 免费 |
| 双盲对比 | AI vs 真人笔记混评 | 中 |
| A/B 线上 | 真实发群对比转化 | 高 |
| 自动指标 | 采纳率、修改率自动上报 | 低 |

**推荐**：自动指标 + 运营评分，先跑起来。

## 三、类目体系方案

### 范围界定

快团团共有 12 个一级类目。**本项目聚焦食品领域，当前阶段只做其中 2 个**：

| 一级类目 | 当前阶段 | 理由 |
|------|:--:|------|
| **食品保健** | ✅ 在做 | 项目核心赛道 |
| **水果生鲜** | ✅ 在做 | 笔记写法与食品接近 |
| 美容个护 | 📋 后期 | 不同品类，需独立模块结构 |
| 服饰箱包 | 📋 后期 | 同上 |
| 数码电器 | 📋 后期 | 同上 |
| 其余 7 个 | 📋 后期 | 同上 |

### 不同一级类目需要不同的模块结构

食品笔记需要"口感、配料、冷链"，服饰笔记需要"面料、尺码、搭配"——不能共用一套 14 模块。

| 一级类目 | 模块数 | 状态 |
|------|:--:|:--:|
| 食品保健 | 14 个（hook/price/taste/trust...） | ✅ 已做 |
| 水果生鲜 | 复用食品 14 模块（改动最小） | 🔜 |
| 美容个护 | ~12 个（钩子/成分/功效/使用感...） | 📋 |
| 服饰箱包 | ~10 个（钩子/款式/面料/尺码...） | 📋 |
| 其余 | 逐个扩展 | 📋 |

**架构复用**：左侧配置、中栏编辑、右侧版本对比、快捷指令、流式输出这套框架不重写，只换模块定义。

### 食品类目内部：品类簇方案

食品保健 + 水果生鲜下面有 7 个二级类目，每个二级下面又有几十个三级类目。不按类目级数做，按**写法像不像**分组。

#### 真实类目 → 品类簇映射

| 二级类目 | 包含三级类目 | 归入哪些簇 |
|------|------|------|
| 咖啡/麦片/冲饮 | 常温乳制品、低温乳制品、成人奶粉、饮料、咖啡、冲饮品、饮用水 | 乳品 + 饮料冲调 |
| 茶 | 各种茶叶 | 饮料冲调 |
| 零食/坚果/特产 | 零食、坚果、肉脯、特产 | 零食膨化 + 坚果干货 + 肉脯卤味 |
| 粮油米面/南北干货/调味品 | 米面、干货、调味品 | 粮油干货 + 调味佐餐 |
| 酒类 | 白酒、红酒、啤酒、果酒 | 酒类 |
| 保健食品/膳食营养 | 保健食品、营养补充 | 滋补保健 |
| 传统滋补品 | 阿胶、枸杞、蜂蜜 | 滋补保健 |

#### 14 个品类簇

| 簇 | 覆盖品类 | 写作特征 | 语料需求 |
|------|------|------|:--:|
| 乳品 | 酸奶、牛奶、奶酪、奶片、奶粉 | 奶源、口感、冷链 | 5-8 篇 |
| 饮料冲调 | 咖啡、茶、冲饮粉、饮用水 | 提神、解暑、便携、冲泡 | 5-8 篇 |
| 酒类 | 白酒、红酒、啤酒、果酒 | 年份、口感、送礼、场景 | 5-8 篇 |
| 零食膨化 | 薯片、虾条、爆米花 | 酥脆、追剧、解馋 | 5-8 篇 |
| 坚果干货 | 坚果、炒货、果干 | 原味、健康、每日 | 5-8 篇 |
| 肉脯卤味 | 肉干、鸭脖、卤蛋 | 解馋、追剧、下酒 | 5-8 篇 |
| 烘焙糕点 | 面包、蛋糕、饼干 | 松软、早餐、下午茶 | 5-8 篇 |
| 糖果巧克力 | 巧克力、糖果、果冻 | 甜蜜、礼物、解压 | 5-8 篇 |
| 速食主食 | 方便面、水饺、螺蛳粉 | 方便、懒人、夜宵 | 5-8 篇 |
| 调味佐餐 | 辣酱、火锅底料 | 下饭、百搭、厨房 | 5-8 篇 |
| 冷冻甜品 | 冰淇淋、雪糕 | 消暑、解馋、怀旧 | 5-8 篇 |
| 生鲜水果 | 水果、蔬菜、鸡蛋 | 产地、时令、新鲜 | 5-8 篇 |
| 粮油干货 | 大米、油、面粉 | 产地、品质、家庭 | 5-8 篇 |
| 滋补保健 | 蜂蜜、枸杞、阿胶、保健食品 | 养生、送礼、功效、成分 | 5-8 篇 |

**总计：14 簇 × 6 篇 = ~80 篇语料。不是几十个三级类目 × N 篇。**

### 簇怎么用

**不改 14 模块结构。** 每个簇有一张配置表，生成时注入对应的"调味料"：

```js
// 生成时查表 → 注入 keywords → 同一个 modules.js 生成不同品类笔记
{
  '乳品':    { keywords: ['拉丝','奶香','冷链'], emphasize: ['taste','trust'] },
  '酒类':    { keywords: ['年份','送礼','场景'], emphasize: ['origin','scene'] },
  '滋补保健': { keywords: ['功效','成分','适宜人群'], emphasize: ['ingredient','faq'] },
}
```

### POC 策略

先做乳品一个簇，从**常温乳制品**的三级类目开始（项目案例"每日吨吨"的实际类目）。5 篇语料，跑通后扩展到低温乳制品、成人奶粉（同属乳品簇，写法差异极小）。

### 语料格式

按 14 模块结构存储，直接复用现有项目格式：

```json
{
  "id": "dairy-001",
  "cluster": "乳品",
  "content": {
    "hook": "🌱云贵山野入瓶！...",
    "price": "💰 团购价：79.9元...",
    "taste": "打开就能闻到..."
  },
  "images": [
    { "file": "img.jpg", "module": "taste", "order": 1, "desc": "拉丝特写" }
  ],
  "metrics": { "likes": 2300 },
  "tags": ["酸奶", "益生菌"]

```

### 运营怎么协作

发一个 Markdown 模板，运营把原文按模块复制粘贴 + 标注图片位置。一篇 10 分钟，5 篇一小时。收到后脚本转 JSON 入库。

详见 [corpus-building-guide.md](corpus-building-guide.md)。

## 五、Agent 实现路径

从工具到 Agent 的四阶段：

| 阶段 | 新增能力 | 核心实现 |
|:--:|------|------|
| 1 | 视觉分析 + 语料检索 | GLM-4V + RAG |
| 2 | 自主决策 | 规则引擎（纯 JS，不调 AI） |
| 3 | 用户记忆 + 学习 | localStorage + 统计 |

**核心洞察**：阶段 2-4 都不需要新模型，全部用规则引擎实现。真正的"Agent 感"来自决策逻辑，不是模型大小。

## 六、模型调优：埋点 + 数据驱动

### 为什么需要埋点

当前系统是"盲飞"状态——不知道运营改了什么、用了什么功能、哪里不满意。埋点让一切可量化。

### 埋什么

| 事件 | 采集字段 | 回答什么问题 |
|------|------|------|
| 一键生成 | 品类、选中模块、风格 | 什么品类生成最多？ |
| 采纳 | 版本 1 还是 2、采纳了哪几个模块 | 哪个版本更受欢迎？ |
| 快捷指令 | 按钮类型、单模块还是全模块 | 运营最常用什么操作？ |
| 编辑 | 改了哪个模块、改动多少字 | AI 哪个模块写得最差？ |
| 导出 Word | 触发频率 | 有多少笔记实际被使用？ |
| 运营评分 | ⭐ 1-5（采纳/导出后弹窗） | 运营主观满意度 |

### 工具选型

| 阶段 | 方案 | 说明 | 费用 |
|:--:|------|------|:--:|
| POC | 服务端记日志 | 后端每操作记一条 JSON，写一个简单统计页 | ¥0，半天 |
| MVP | **Umami**（推荐） | 免费开源，自己部署在 Render 上，一个命令启动，自带 Dashboard，数据不外传 | ¥0 |
| 进阶 | **PostHog** | Session 回放 + 用户路径 + 漏斗分析，可看运营完整操作录屏 | 免费（100 万事件/月） |
| 国内备选 | 神策数据 | 国内访问快，专门做用户行为分析 | 有免费版 |

**推荐路线**：POC 记日志 → MVP 自部署 Umami（轻量、私有、不花钱）→ 需要看 session 回放时切 PostHog。

### 调优闭环

```
每两周拉一次埋点数据
        │
        ▼
看哪个模块编辑率最高
        │
        ▼
优先优化该模块的 prompt
        │
        ▼
再跑两周 → 编辑率下降了吗？
        │
        ▼
下降了 → 优化有效 → 记录 → 下一个模块
没下降 → 换方案 → 继续迭代
```

不是"调一次就完"，是**持续监控 + 持续迭代**。
