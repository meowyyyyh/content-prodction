## 整体架构

```
                        用户浏览器
                   https://kuaigao.xxx.com
                            │
                            ▼
                   Cloudflare Pages
                    静态前端托管
                    国内访问友好
                            │ /api/*
                            ▼
                   Render / Railway
                   Express 后端
                            │
          ┌────────┬────────┼────────┬────────┐
          ▼        ▼        ▼        ▼        ▼
      DeepSeek  GLM-4V   语料库   合规扫描  用户画像
      (文本)   (视觉)   (RAG)   (关键词)  (localStorage/SQLite)
                            │
                            ▼
                    编排器 (Orchestrator)
                    纯 JS 规则引擎

```

## 核心模块

### 1. 编排器（Orchestrator）— Agent 的大脑

```
用户：帮我写一篇酸奶笔记，图片已上传

Orchestrator：
  │
  ├─ Step 1: 分析输入
  │    ├─ 品类识别：酸奶 → 乳品簇
  │    └─ 图片分析：3 张图 → 标签 + 描述（GLM-4V）
  │
  ├─ Step 2: 检索参考
  │    └─ 语料库检索：找到 3 篇乳品高分笔记（RAG）
  │
  ├─ Step 3: 决策
  │    ├─ 风格：用户偏好 闺蜜风 → 用闺蜜风模板
  │    ├─ 长度：用户历史偏好详细 → 不压缩
  │    └─ 图片位置：口感图→taste顶部，场景图→scene底部
  │
  ├─ Step 4: 生成
  │    └─ 组装 prompt → DeepSeek → 14 模块文案
  │
  ├─ Step 5: 自检
  │    ├─ 合规扫描、字数检查、标题污染、emoji 密度
  │    └─ 有问题 → 自动修复
  │
  └─ Step 6: 输出 + 记住偏好
```

**实现**：一个函数，用规则引擎，不调 AI 做决策。

### 2. 工具层

| 工具 | 模型/方式 | 作用 | 费用 |
|------|------|------|:--:|
| 文本生成 | DeepSeek | 写 14 模块文案 | 按量 |
| 视觉分析 | GLM-4V（待定） | 看图→分类+描述 | 免费 |
| 语料检索 | DeepSeek embedding | 搜高分笔记 | 按量 |
| 合规扫描 | 关键词匹配 | 违禁词检测 | 免费 |

## 数据流（一次完整生成）

```
用户：填商品 + 上传 3 张图
     │ POST /api/generate
     ▼
orchestrator：接收任务
     │
     ├──→ vision.js      分析图片 → 标签 + 描述
     ├──→ rag.js         检索乳品语料 → 3 篇参考
     ├──→ profile.js     读取用户偏好
     │
     ├──→ orchestrator   综合决策：风格、长度、图片位置
     │
     ├──→ generator.js   组装 prompt → DeepSeek → 14 模块
     │
     ├──→ compliance.js  扫描违禁词
     ├──→ orchestrator   自检：合规、字数、标题、emoji
     │
     ├──→ profile.js     更新画像
     │
     └──→ 返回前端       14 模块 + 图片 + 元数据
```

## 护城河在架构中的体现

### 这三层架构，每一层都在加厚护城河

```
┌─────────────────────────────────────────────────────────┐
│                   第三层：合规知识库                        │
│  每个品类簇的违禁词、每个平台的规则、每次踩坑的记录           │
│  别人抄不走 —— 这是时间堆出来的经验                        │
├─────────────────────────────────────────────────────────┤
│                   第二层：数据飞轮                          │
│  用户画像 → 编辑模式 → 采纳率 → 每次修改都在训练系统          │
│  别人抄不走 —— 你没有这 1000 次生成的数据                    │
├─────────────────────────────────────────────────────────┤
│                   第一层：工作流深度                          │
│  商品录入 → 生成 → 配图 → 审核 → 发布 → 效果反馈            │
│  别人能抄功能，但抄不了这套 SOP 嵌入你业务的程度              │
└─────────────────────────────────────────────────────────┘
```

### 具体价值测量

#### 效率指标

| 指标 | 当前状态 | AI 辅助后（目标） | 测量方式 |
|------|:--:|:--:|------|
| 初稿生成时间 | 人工写稿 30min | AI 10s 生成 | 自动计时 |
| 单篇总耗时 | 写稿 30min + 排版 30min | AI 10s + 修改 5min | 埋点统计 |
| 每人每天产能 | 5-8 篇 | 15-20 篇 | 按天统计 |
| 新运营上手时间 | 2-4 周 | 1 天（有 AI 辅助） | 主观评估 |

#### 质量指标

| 指标 | 当前状态 | AI 辅助后（目标） | 测量方式 |
|------|:--:|:--:|------|
| 风格一致性 | 不同运营写出来不同 | 模板兜底，调性统一 | 抽查对比 |
| 合规返工率 | 审出问题返工 | 初稿已过合规 | 埋点统计 |
| 采纳率 | — | > 70%（运营直接用的比例） | 埋点统计 |
| 修改率 | — | < 30% 字数改动 | 埋点统计 |

#### 业务价值

```
人力成本节省：如果每人每天产能从 8 篇提升到 20 篇，
             同样的工作量只需要不到一半的人。

质量稳定性：AI 不会因为"今天状态不好"写出烂稿。

响应速度：上午给商品资料，下午就能出稿。
```

### 技术路线的"护城河友好性"评估

每次做技术决策时，可以问自己一个问题：
**"这个决定是在加厚护城河，还是在加速被淘汰？"**

| 决策 | 护城河友好 | 原因 |
|------|:--:|------|
| 用 DeepSeek 而非自研模型 | ✅ | 模型能力交给专业公司，你把精力放在数据和流程上 |
| 纯规则引擎做 Agent 决策 | ✅ | 可控、可预测、不依赖第三方 API，换模型不换逻辑 |
| 语料库结构化存储 | ✅ | 数据是唯一不会贬值的资产 |
| 用户画像本地存储 | ✅ | 用户数据不外传，也是信任壁垒 |
| 用 vision 模型而非规则解析图片 | ⚠️ | 依赖第三方 API，但有足够备选方案 |
| 做全品类扩展而不是深挖一个品类 | ❌ | 表面看起来功能多，但每个品类都不够深，数据飞轮转不起来 |
