# 多模态聊天方案：Doubao-Seed-2.0-mini 驱动图片排版

> 版本：1.0 · 日期：2026-07-22 · 状态：待评审

---

## 1. 背景

### 1.1 当前文本方案的固有问题

| 问题 | 根因 |
|------|------|
| 图片标记消失 | AI 不理解 `[IMG:N]` 标记的语义，可能删除或篡改 |
| 无法按视觉逻辑排版 | AI 只能读 alt 文本，看不到图，无法判断"正面→打开→内容"的顺序 |
| 图片跨模块迁移 | 只能靠文本指令，AI 无法确认目标模块 |
| 描述文本污染 | 图片 alt 存了 400 字 vision 描述，混入文案 |

### 1.2 目标

中栏聊天框接入多模态模型，模型**真正看到图片**，实现：

- 图片不会消失（视觉关联强）
- 乱序图片按视觉逻辑排版（"从展示到打开"）
- 图片跨模块迁移（"把这张图放到 taste 模块"）
- 根据图片内容生成精准文案

## 3. 技术设计

### 3.1 API 调用方式

使用现有 Volcengine Ark 端点（和 vision.js 相同配置），消息格式为多模态：

```javascript
// 新端点：POST /api/chat/multimodal (SSE 流式)
{
  "messages": [
    {
      "role": "system",
      "content": "你是文案改写+图片排版助手..."
    },
    {
      "role": "user",
      "content": [
        { "type": "image_url", "image_url": { "url": "data:image/jpeg;base64,..." } },
        { "type": "image_url", "image_url": { "url": "data:image/jpeg;base64,..." } },
        { "type": "text", "text": "## 当前文案\n...\n## 用户指令\n把图片按展示到打开的顺序排版" }
      ]
    }
  ]
}
```

### 3.2 图片发送策略

| 场景 | 分辨率 | 单张大小 | 10张 | 预期延迟 |
|------|--------|----------|------|----------|
| 只看排版关系 | 128px | 3-5KB | 30-50KB | 3-5秒 |
| 需要看文字 | 256px | 10-15KB | 100-150KB | 5-10秒 |
| 需要看细节 | 512px | 30-50KB | 300-500KB | 15-30秒 |

**默认 256px**——够看清产品结构（正面/打开/内容），延迟可接受。

### 3.3 AI 输出格式

两种标记同时输出，各司其职：

**`[IMAGE:N]` 内联标记**——表示文本中图片的精确位置：

```
先看外包装，亮绿色袋子印着品牌名
[IMAGE:0]
打开后薯片金黄酥脆
[IMAGE:1] [IMAGE:2]
两袋放一起对比
```

**`---LAYOUT---` JSON 块**——跨模块迁移和排序指令：

```json
---LAYOUT---
{
  "0": {"module": "hook", "before": "段落1"},
  "1": {"module": "taste", "group": "2"},
  "2": {"module": "taste", "group": "1"}
}
```

JSON 只包含有变化的图片（不动的不写）。

### 3.4 System Prompt 设计

```
你是电商笔记文案改写+图片排版助手。给你当前模块文案、指令、和几张商品图片。

## 图片引用规则
1. [IMAGE:N] 是图片占位符（N从0开始），对应第N张输入图片
2. 在文案中自然穿插 [IMAGE:N]，表示该位置插入对应图片
3. 多图并列：[IMAGE:0] [IMAGE:1] 连续写，表示图片组
4. 一个模块内不超20个标记，不增删

## 排版规则
1. 用户说"图文穿插"→ 根据图片角色放到相关文字旁
2. 用户说"按XX顺序排"→ 看图判断图片属于哪个阶段，排好顺序
3. 需要跨模块移图→ 文本末尾加 ---LAYOUT--- JSON

## 跨模块迁移（---LAYOUT---）
当图片应放到其他模块时，在文本末尾加 JSON：
{
  "N": {"module": "目标模块key"}
}
- module 可选：hook/price/taste/trust/aftercare/tips/cta/ingredient/origin/brand/scene/feedback/faq
- 不需要移动的图片不写
```

### 3.5 前端解析

```javascript
// 从 AI 响应中提取文本和排版指令
function parseMultimodalResponse(rawText) {
  const layoutMatch = rawText.match(/---LAYOUT---\s*(\{[\s\S]*?\})\s*$/)
  let layout = {}
  if (layoutMatch) {
    try { layout = JSON.parse(layoutMatch[1]) } catch {}
    rawText = rawText.replace(/---LAYOUT---[\s\S]*$/, '').trim()
  }
  
  // 文本中的 [IMAGE:N] 在打字完成后替换为真实图片
  return { text: rawText, layout }
}
```

### 4.2 兼容性

| 场景 | 行为 |
|------|------|
| 无图 + 聊天/快捷按钮 | 走现有文本端点（DeepSeek），不变 |
| 有图 + 聊天/快捷按钮 | 走 multimodal 端点（Doubao） |
| 图片只在部分模块 | 只发有图的模块的图片 |
| 多模态调用失败 | 降级到现有文本端点 + toast 提示 |

### 4.3 退化路径

```
检测有图？
  ├─ 无图 → 走现有 /api/chat/stream (DeepSeek)
  └─ 有图 → 走 /api/chat/multimodal (Doubao)
             ├─ 成功 → 解析文本 + 排版指令
             └─ 失败 → 降级到现有文本方案 + toast
```

## 5. 风险与对策

| 风险 | 对策 |
|------|------|
| 多模态 API 不稳定 | 自动降级到文本方案 |
| 大量图片导致超时 | 限制单次最多20张；超20张时提示用户分批 |
| `---LAYOUT---` 解析失败 | JSON parse 异常时忽略，只应用文本标记 |
| 模型对 `[IMAGE:N]` 的使用不稳定 | 在 prompt 中强化规则；前端校验标记数量 |
| 图片跨模块迁移后 state 不一致 | `onConfirm` 统一更新 classifiedImages |
