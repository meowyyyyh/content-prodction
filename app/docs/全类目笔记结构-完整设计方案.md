## 一、架构决策

### 1.1 模块三层模型

```
Layer 1: 通用模块（5个）→ 所有实体商品类目共享同一份 prompt
Layer 2: 类目适配模块（5个）→ 共享 key，prompt 按 Tier A/B/C 三档分化
Layer 3: 类目专属模块（N个）→ 每个类目独有的模块 key + prompt
```

### 1.2 通用模块（Layer 1 — 不改）

| key | label | 原因 |
|-----|-------|------|
| `hook` | 首屏钩子 | 开场结构通用，prompt 不耦合特定品类 |
| `price` | 价格福利 | 算账逻辑通用 |
| `cta` | 行动召唤 | 下单引导通用 |
| `aftercare` | 物流售后 | 物流发货通用 |
| `faq` | 常见问题 | Q&A 格式通用 |

### 1.3 类目适配模块（Layer 2 — 共享 key，prompt 按 Tier 分化）

**核心改进**（合入审核反馈 #4）：不是 14 个类目各写一份，而是合并为三档：

| Tier | 覆盖类目 | 分化策略 |
|------|------|------|
| **Tier A（强分化）** | 美食酒水、美妆洗护、母婴亲子、医药健康 | 各自独立 prompt — 信任维度、种草逻辑确实不同 |
| **Tier B（合并共享）** | 数码家电+办公学习、家具+家装建材、汽车旅行、鞋包服饰 | 组内共享一份 prompt — 核心结构相同，只需类目名变量注入 |
| **Tier C（通用兜底）** | 其余类目（居家生活、餐厨用品、文体健康、钟表眼镜） | 使用 `__universal__` 兜底 prompt |

5 个适配模块 × 4 档（Tier A 4个 + Tier B 3组 + Tier C 统一 + 虚拟卡券特殊）= **减少约 30 个 prompt**，降低维护成本。

| key | label | 为什么需要分化 |
|-----|-------|------|
| `trust` | 基础信任 | 食品靠配料表、数码靠参数、服饰靠面料、美妆靠成分安全 |
| `brand` | 品牌背书 | 食品讲牧场/奶源、数码讲技术积累、服饰讲设计师 |
| `scene` | 场景共情 | 食品讲吃/w场合、服饰讲穿/w场合、数码讲用/w场合 |
| `tips` | 使用贴士 | 食品=储存食用、数码=维护技巧、服饰=洗护保养 |
| `feedback` | 用户反馈 | 不同品类用户画像和评价维度完全不同 |

**Prompt 查找逻辑**：`MODULE_CONTENT_REQS[catLevel1]?.[moduleKey] ?? MODULE_CONTENT_REQS.__universal__[moduleKey]`

### 1.4 模块 key 命名规范

- 已有 food 模块保留原名：`taste`、`ingredient`、`origin`
- 新增专属模块用语义化命名：`texture`、`efficacy`、`fabric`、`safety` 等
- ⚠️ 避免与已存在/已删除的 key 冲突（如 `comparison` 历史上是"全网比价"，数码家电的"横向对比"改为 `product_compare`）
- ModuleKey 联合类型随每批实施递增

### 1.5 已有代码清理（合入审核反馈 #2）

`moduleRegistry.ts` 的 DEFAULT_MODULE_ORDER 中美妆洗护残留了旧 key（`beauty_effect`、`beauty_ingredient`、`usage`），这些是 Phase 3 TODO 时期留下的占位符，从未在 ModuleKey 类型中正式定义过。实施时统一替换为方案中的新 key。

### 类目 2：美妆洗护 🆕 第 1 批

**新增专属模块（6个）**：

| key | label | description |
|-----|-------|------|
| `texture` | 质地体验 | 质地、肤感、吸收度、气味 |
| `efficacy` | 功效解析 | 核心成分功效、使用效果、作用机理 |
| `usage_method` | 使用方法 | 正确步骤、用量、早晚使用区别、搭配建议 |
| `before_after` | 效果对比 | 使用前后变化（不夸大、基于事实）⚠️ 禁止量化功效承诺（"7天变白""皱纹消失"等） |
| `ingredient_analysis` | 成分解读 | 明星成分科普、浓度/配方分析 |
| `suitable_skin` | 适用肤质 | 不同肤质适用性、敏感肌测试结果 |

**适配模块 prompt**（Tier A — 强分化）：
- `trust` → 成分安全性+第三方检测+备案号/批准文号
- `brand` → 实验室背景/研发团队/专利技术（不是牧场规模）
- `scene` → 护肤场景（晨间/晚间/换季/妆前/熬夜急救）
- `tips` → 使用频率+开封保质期+搭配禁忌+储存方法
- `feedback` → 按肤质分类的用户评价（干皮/油皮/混油/敏感肌）

**模块配置**：
```
Mandatory:  hook, price, cta
Recommended: texture, efficacy, ingredient_analysis, usage_method, suitable_skin, trust, brand, scene, tips, faq
Optional:    before_after, feedback, aftercare
```

**DEFAULT_MODULE_ORDER**：
```
hook → price → texture → efficacy → ingredient_analysis → usage_method → suitable_skin → trust → brand → before_after → scene → tips → feedback → faq → aftercare → cta
```

**实施注意**：同步清理 `moduleRegistry.ts` 中残留的 `beauty_effect`/`beauty_ingredient`/`usage` 旧 key 引用。

### 类目 14：医药健康 ⚠️ 合规保守方案（第 2 批，提前实施）

**设计原则**：
- 所有模块采用「产品说明书」风格，客观描述，禁止渲染感受
- 禁止模块：`scene`（场景共情）、`feedback`（用户反馈）、`hook`（首屏钩子涉及亲测）
- 区分 OTC vs 处方药：处方药严禁任何形式广告，OTC 可做有限的客观描述
- UI 在类目选择为医药健康时显示红色合规提示条

**⚠️ 全局合规注入 COMPLIANCE_INJECTION**（合入审核反馈 #5）：

以下规则注入所有模块的 system prompt，优先级最高，违反将导致输出作废：

```
⚠️ 合规规则（医药健康类目，最高优先级）：
- 禁止在任何模块中使用"治愈""根治""奇效""亲测""推荐"
- 禁止任何形式的用户评价、使用体验、使用场景渲染
- 所有功效描述必须引用批准文号对应的说明书内容
- 本品为OTC药品，请按说明书使用或在药师指导下购买和使用
- 禁止价格比较、限时优惠、库存紧张等促销表达
```

**price 模块特殊处理**：
- 药品广告法禁止"优惠""特价""促销"等价格诱导
- price 模块只客观罗列价格信息（规格对应的价格），禁止价格比较和优惠表达

**cta 模块特殊处理**：
- 行动召唤不写"下方直接购买"，改为"如需购买请咨询药师"或"遵医嘱使用"
- 禁止限时、限量、紧迫感等促销话术

**专属模块（4个）**：

| key | label | description |
|-----|-------|------|
| `product_info` | 产品信息 | 品名、规格、成分、批准文号（替代 hook 作为首屏） |
| `usage_dosage` | 用法用量 | 用法说明、剂量、频次、疗程 |
| `precautions` | 注意事项 | 禁忌人群（首段强制）、不良反应、药物相互作用 |
| `qualification` | 资质证照 | 批准文号、生产许可、GMP 认证 |

**适配模块 prompt**（Tier A — 强分化）：
- `trust` → 资质认证+批准文号+药企背景（禁止用户推荐/亲测）
- `brand` → 药企研发背景+生产标准+行业地位
- `tips` → 储存条件+使用提醒+有效期（禁止使用建议外内容）
- `faq` → 常见用药咨询（客观回答，禁止暗示选择倾向）

**模块配置**：
```
Mandatory:  product_info, precautions, cta
Recommended: usage_dosage, trust, qualification, brand, faq
Optional:    price, tips, aftercare
```
> 注：price 从 mandatory 降为 recommended（部分处方药场景不需要价格模块）

**禁用词规则**（写入合规检查 + generator prompt）：
- ❌ 禁止："治愈""根治""亲测有效""推荐使用""效果很好""神奇"
- ❌ 禁止：暗示疗效、对比竞品药物、患者案例
- ❌ 禁止：scene 场景共情（"头疼了一天，吃了这个药……"）
- ❌ 禁止：价格比较、优惠促销表达
- ✅ 允许：客观成分说明、批准文号引用、说明书内容转述

**DEFAULT_MODULE_ORDER**：
```
product_info → usage_dosage → precautions → qualification → trust → brand → price → tips → faq → aftercare → cta
```
> 注：price 在医药类目放靠后位置（用户看完产品信息+用法+禁忌+资质后，价格才是次要决策因素）

## 六、prompt 查找逻辑变更

```js
// 旧（generator.js）
const req = MODULE_CONTENT_REQS[moduleKey]

// 新
function getModulePrompt(moduleKey, catLevel1) {
  // ① 类目专属 prompt
  const catPrompts = MODULE_CONTENT_REQS[catLevel1]
  if (catPrompts?.[moduleKey]) return catPrompts[moduleKey]
  // ② 通用 prompt
  if (MODULE_CONTENT_REQS.__universal__?.[moduleKey]) return MODULE_CONTENT_REQS.__universal__[moduleKey]
  // ③ 兜底（跨类目共享 key，如 taste 只有 food 有）
  return null
}

// 医药类目额外注入
if (catLevel1 === '医药健康') {
  systemPrompt = COMPLIANCE_INJECTION + '\n\n' + systemPrompt
}
```
