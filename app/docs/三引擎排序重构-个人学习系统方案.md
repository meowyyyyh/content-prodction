### 三版排序来源

| 版本 | 标签 | 排序来源 | 风格由谁决定 |
|------|------|------|------|
| V1 | 默认风格 | **用户学习** | 固定位置，个人进化 |
| V2 | Elo #1 风格 | **语料库参考** | Elo 排名第一的纯风格（冷启动：xiaohongshu） |
| V3 | Elo #2 风格 | **语料库参考** | Elo 排名第二的纯风格（冷启动：fun） |

### 全6种风格统一逻辑

6种纯风格（xiaohongshu / girlfriend / fun / minimalist / premium / senior）全部走相同的语料参考逻辑：

```
corpusReferenceOrder(category, style)
  → 查 /api/corpus/search?style={style}
  → 有匹配语料 → 取最高分/最新语料的 moduleOrder（评分相同时按时间倒序）
  → 无匹配语料 → 静默回退 DEFAULT_MODULE_ORDER
```

V2/V3 的具体风格由 Elo 排名动态决定（冷启动阶段使用静态默认值），不是写死的。哪个风格排在前面，就用哪个风格的语料。

**冷启动语料覆盖预期**：当前仅有 xiaohongshu 风格 1 篇语料。其他 5 种风格在语料积累前将始终回退到 DEFAULT_MODULE_ORDER——这是预期行为，随着运营导出逐步覆盖。

## 四、三版生成：各自取序

### 4.1 生成流程

采用**先取序再生成**策略：

```
用户点「一键生成」
  → 确定 V2/V3 风格：
      读 localStorage['flywheel_scores:{userId}'].style_elos
        → 存在且有效 → Elo 排名取 top 2
        → 不存在 → 用 DEFAULT_INPUT.versionStyles（冷启动：V2=xiaohongshu, V3=fun）
  → 取 input.selectedModules
  → 并行计算三个版本的 orderedKeys：
      V1 = userLearnedOrder(catLevel1, catLevel2, catLevel3)                    // 同步
      V2 = await corpusReferenceOrder(catLevel1, catLevel2, catLevel3, v2Style) // 异步
      V3 = await corpusReferenceOrder(catLevel1, catLevel2, catLevel3, v3Style) // 异步
  → 各版按各自顺序过滤 selectedModules 得到 orderedKeys
  → 并行触发三个 streamGenerate(product, orderedKeys_v1, ...)
      streamGenerate(product, orderedKeys_v2, ...)
      streamGenerate(product, orderedKeys_v3, ...)
```

**注意**：Step 2 的 `corpusReferenceOrder()` stub 必须返回 `Promise<ModuleKey[]>`（async 函数），确保 Step 5 替换为真实实现时调用点无需改动。

### 4.2 用户学习顺序 `userLearnedOrder()`

三层查找链（与 moduleRegistry 同构）：

```
① __catLevel1__catLevel2__catLevel3__ → 三级精确
② __catLevel1__catLevel2__            → 同二级下聚合
③ __catLevel1__                       → 同一级下聚合
④ DEFAULT_MODULE_ORDER[catLevel1] || DEFAULT_MODULE_ORDER['__default__']
```

- 聚合算法：按使用频次加权排序，未出现的模块按 DEFAULT_MODULE_ORDER 补末尾
- 独立阈值：≥ 10 次导出后完全独立，不再继承父级

### 4.3 语料参考顺序 `corpusReferenceOrder()`

适用全部6种风格。搜索 API 服务端三层回退：

```
GET /api/corpus/search?catLevel1=&catLevel2=&catLevel3=&style=&top=3

服务端：
  ① level3 精确匹配 → 返回
  ② 回退 level2（同二级下语料）
  ③ 回退 level1（同一级下语料）
  ④ 返回空 → 前端兜底 DEFAULT_MODULE_ORDER

前端处理：
  ① 搜索结果已按动态评分（质量×验证×时效）降序排列，详见《语料动态评分系统设计方案》
  ② 取 Top-3 加权随机选一篇语料（详见动态评分方案 §四）
  ③ 取选中语料的 moduleOrder
  ④ 过滤掉不在 selectedModules 中的模块
  ⑤ 补充 selectedModules 中但语料没有的模块（按 DEFAULT_MODULE_ORDER）
  ⑥ 搜索失败/无结果 → 静默回退（不弹提示）
```

搜索目录：`data/corpus-review/`。

**历史语料兼容**：搜索 API 遇到缺少 `moduleOrder` 字段的旧语料时，跳过该条目（不报错），继续查下一条。缺少 `editRate`/`moduleOrderOriginal` 字段时使用默认值（editRate→0.3，moduleOrderOriginal→false）。

## 五、学习系统：导出时采集 + 分析

### 5.1 学习时机

所有学习在**导出时**触发（异步，不阻塞导出）。

### 5.2 学习维度（5维度）

**维度 1：模块顺序**

```js
// localStorage['user_learned_order:{userId}']
{ "美食酒水::酒水饮料::乳制品": { "orders": [[...], ...], "count": 23 } }
```

**维度 2：图片排版偏好（Phase 1）**

| 子维度 | Phase |
|------|:--:|
| 图片-文字顺序变化 | 1 |
| 图片数量增减 | 1 |
| 图片跨模块移动 | 2 |
| 图组布局变化 | 2 |

前置条件：`buildCorpusJSON` 中新增 `sourceImageId`。

**维度 3-5**：风格偏好 / 文风句式 / 编辑距离 — 已在数据飞轮方案中完整设计。

### 5.3 采纳快照机制

```typescript
interface AdoptionSnapshot {
  moduleKey: string
  aiContentHash: string        // AI 原文 hash，不存全文
  aiImageIds: string[]          // 图片 ID，不存 base64
  imagePositions: Record<string, 'before_text' | 'after_text' | 'interleave'>
  adoptedAt: number
  sourceVersion: 1 | 2 | 3
  sourceStyle: ContentStyle
}
// localStorage['adoption_snapshot:{userId}']，容量 < 5KB
```

**⚠️ 关键：写入顺序（连续采纳安全）**

```
handleAdopt(moduleKey, content, images):
  1. 先读取当前 centerModules 状态
  2. 对该模块写快照（从当前 centerModules 中取该模块的 AI 原文 hash + 图片 ID）
  3. 将快照合并入 adoption_snapshot（用 moduleKey 索引，不覆盖其他模块的快照）
  4. 再调用 setCenterModules 修改内容
```

这样连续采纳模块 A、B、C 时，每个模块的快照互不覆盖，导出时能拿到完整的多模块快照。

**清除时机**：导出时读取对比后清除。
