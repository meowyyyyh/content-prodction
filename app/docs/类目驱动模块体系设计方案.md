# 类目驱动模块体系设计方案 v3

> 2026-07-19 · 基于类目.xlsx的新三级类目体系 · v3 合入Codex两轮审阅反馈
> 原则：不同类目 → 不同模块体系 → 不同文案风格 → 不同排版模式

---

## 一、设计目标

1. **运营选类目 → 系统自动推荐模块组**（运营可增减）
2. **图片分析补充建议**（doubao识别到配料表 → 自动勾选ingredient）
3. **不该出现的模块不出现**（数码类目下没有"口感体验"）
4. **通用模块跨类目复用**（hook/price/trust等一次写好，所有类目共享）
5. **专属模块各自独立**（taste只有美食酒水有，specs只有数码家电有）
6. **数据飞轮跟着类目走**（同类目信号汇聚，通用模块跨类目共享学习）

## 三、类目 → 默认模块映射

### 3.1 推荐流程

```
运营选三级类目（catCode）
        ↓
查 CATEGORY_MODULE_MAP[catCode] → 默认模块列表（部分已勾选）
        ↓
doubao 分析图片 → suggestedModule 命中可选模块 → 自动勾选
        ↓
运营手动增减模块 → 最终确认
        ↓
按类目默认顺序排列模块
```

### 3.2 映射规则

- **mandatory**：始终默认勾选、运营不可取消。当前设为 hook/price/cta — 这三者是所有商品笔记的骨架（开场→报价→收尾），缺少任何一个会导致笔记结构不完整。后续可按类目调整 mandatory 范围。
- **recommended**：默认勾选、运营可取消（类目推荐模块）
- **optional**：默认不勾选、运营可手动加、图片分析可触发自动勾选（如生鲜类目的 ingredient）

### 3.3 美食酒水映射表

**A组 — 加工食品（基准配置）：**

适用三级类目：乳制品(5543)、冲饮茗茶(468-478)、酒水饮料(488-500)、休闲美食(447-457)、坚果蜜饯(458-466)

```
mandatory:  hook, price, cta
recommended: taste, trust, ingredient, origin, brand, scene, aftercare, tips, faq
optional:   feedback, comparison
```

**A'组 — 粮油调味（基准但ingredient默认不勾选）：**

适用三级类目：粮油调味(501-508)

```
mandatory:  hook, price, cta
recommended: taste, trust, origin, brand, scene, aftercare, tips, faq
optional:   ingredient, feedback, comparison
```

> 大米/食用油/面粉无配料表，ingredient 默认不勾选。doubao 识别到配料表图片可触发自动勾选。

**B组 — 生鲜食材：**

适用三级类目：水果蔬菜(509-517)、肉蛋海鲜(518-522)

```
mandatory:  hook, price, cta
recommended: taste, trust, origin, brand, scene, aftercare, tips, faq
optional:   ingredient, feedback, comparison
```

> B组 ingredient 默认不勾选。但如果doubao识别到图片中有配料表/营养成分表，自动勾选。

**C组 — 滋补保健：**

适用三级类目：滋补保健(479-487)

```
mandatory:  hook, price, cta
recommended: taste, trust, origin, brand, scene, aftercare, tips, faq
optional:   ingredient, feedback, comparison
```

> C组同B组逻辑。滋补品的 ingredient 偏向"成分/营养"，非"配料表"。

### 3.4 三层查找链

不用逐三级硬编码500+编码。采用"一级默认 → 二级覆盖 → 三级精确覆盖"的三层查找链：

```
getModuleConfig(catCode):
  ① catCode 在 CATEGORY_MODULE_MAP 中有显式配置？
    → 是 → 直接用（处理个别例外，如粮油调味的特定编码）
  ② catCode 所属二级类目在二级例外表中有配置？
    → 是 → 使用该二级配置（处理整个二级类目的例外，如水果蔬菜全算B组）
  ③ catCode 所属一级类目在 LEVEL1_DEFAULTS 中有配置？
    → 是 → 使用该一级默认（如 __food_default__ → A组基准）
  ④ 回退到 __default__
```

**实现：**

```ts

    optional: ['ingredient','feedback','comparison'],
  },
  '__food_meat_seafood__': {  // 美食酒水 > 肉蛋海鲜 → B组
    mandatory: ['hook','price','cta'],

    mandatory: ['hook','price','cta'],
```

### 3.5 图片分析 → 自动勾选逻辑

```
doubao分类结果中 suggestedModule 字段命中 optional 列表中的模块
  → 自动将该模块从 optional 提升为 recommended（勾选）
  → 弹窗提示运营"检测到配料表图片，已自动勾选「成分科普」模块"

如果运营手动取消后又后悔 → 手动勾回来即可
```

> 这个逻辑已经在 `handleConfirmImages` 中部分实现——当前按照 `IMAGE_MODULE_MAP[img.type]` 推荐模块。升级为按 `suggestedModule` + CATEGORY_MODULE_MAP 的 optional 列表做交集。

## 四、前端改动

### 4.1 新类目选择器（三级联动）

数据来源：类目.xlsx 的 `code | level1 | level2 | level3` 三列。三级用 xlsx 中的数字 code（如 `5543`）。一二级用人工命名的稳定编码，规则：一级 = 拼音缩写（如 `food`、`beauty`），二级 = `一级_二级拼音`（如 `food_drinks`、`food_snacks`）。编码一旦确定不再修改——类目名称和层级可能随业务调整，编码作为飞轮数据的持久化 key 必须稳定。新增类目时追加新编码，不改旧编码。

```ts
// src/data/categories.ts 新结构
export interface CategoryNode {
  code: string       // 类目编码
  name: string       // 类目名称
  children?: CategoryNode[]
}

// 从类目.xlsx 编译生成
export const CATEGORY_TREE: CategoryNode[] = [
  {
    code: 'food', name: '美食酒水',
    children: [
      { code: 'food_drinks', name: '酒水饮料', children: [
        { code: '5543', name: '乳制品' },
        { code: '488', name: '白酒' },
        // ...
      ]},
      { code: 'food_snacks', name: '休闲美食', children: [...] },
      // ... 其他二级
    ]
  },
  // P0 只启用美食酒水，其他一级类目置 disabled: true
]
```

### 4.2 LeftPanel 改动

- 三级联动下拉（一级 → 二级 → 三级）
- 选好三级类目 → 读 CATEGORY_MODULE_MAP → 渲染模块勾选列表：
  - mandatory模块：灰色已勾选，不可取消
  - recommended模块：默认勾选，可取消
  - optional模块：默认不勾选，可手动加
  - 其他类目专属模块：不显示
- doubao分析完成后 → 检查 suggestedModule → 自动勾选匹配的 optional 模块

### 4.3 向后兼容

```ts

  // 保留字段（展示用，从 CATEGORY_TREE 反向查找）
```

**flywheel_scores 兼容方案**：旧 key 为品类路径名（如`"调制乳/风味牛奶"`），新 key 为类目编码（如`"5543"`）。读取时先查新 key，无数据则查旧 key 作为冷启动兜底。新信号只写新 key。Phase 1 完成后旧数据可手动迁移或自然淘汰。

## 五、后端改动

### 5.1 generator.js

- `buildPrompt` 接受 ModuleKey[]，按 key 查 `MODULE_CONTENT_REQS` 生成模块指令
- 语料因子分析（`analyzeCorpusFactors`）按 `catCode + moduleKey` 检索
- `ingredient` 模块的 prompt 按类目组调整：A组→配料表分析，B/C组→营养成分亮点

```ts
const MODULE_CONTENT_REQS: Record<string, string> = {
  // === 准通用 ===
  hook: '...', price: '...', trust: '...', brand: '...', scene: '...',
  aftercare: '...', tips: '...', cta: '...', feedback: '...', comparison: '...', faq: '...',
  
  // === 美食酒水 ===
  taste: '描述产品的口感、风味、质地...',
  ingredient: '解读产品的配料表和营养成分...',
  origin: '溯源产品的产地、原料来源...',
  
  // === 美妆洗护（Phase 3） ===
  beauty_effect: '请描述产品的核心功效...',
  beauty_ingredient: '请解析产品核心成分...',
  usage: '请说明产品的使用方法和步骤...',
  
  // === 以下为 TODO，对应 Phase 启动时实现 ===
  // specs / feature / unboxing — Phase 数码家电
  // material / fit / craft — Phase 鞋包服饰
  // safety / age_fit / parenting — Phase 母婴亲子
  // freshness / seasonality — Phase 生鲜果蔬
}
```

### 5.2 语料库检索升级

- 入库时记录 `categoryCode`（三级类目编码）
- 检索回退策略：本级语料≥3篇 → 直接用；不足 → 上溯二级类目；再不足 → 上溯一级；全无 → 回退默认prompt
- 回退时标记 `confidence: 'high' | 'medium' | 'low' | 'fallback'`，注入prompt时附带置信度提示

## 六、数据飞轮关联

### 6.1 信号层级

| 层级 | 粒度 | 说明 |
|---|---|---|
| 三级类目 | 最细 | 如"5543"的taste模块编辑率 |
| 二级类目 | 回退 | 三级数据不足时上溯 |
| 通用模块 | 跨类目 | hook/price等所有类目共享学习 |

### 6.2 数据结构

```js
// flywheel_scores
{
  "style_elos": { /* 不变，跨类目共享 */ },
  
  "module_events": {
    "5543": {  // 类目编码
      taste: { adopted: 45, disliked: 3 },
      hook: { adopted: 42, disliked: 0 },
    }
  },
  
  "module_deletion_events": {
    // 模块被中栏删除 → 推荐质量反馈信号
    "5543": { ingredient: 2, comparison: 1 }
  }
}
```

### 6.3 风格Elo多类目观察

风格Elo当前跨类目共享。当运营活跃在≥3个不同一级类目时，启动观察指标看跨类目Elo是否出现噪音信号。如果出现明显矛盾（同一风格在乳品高分、在数码低分），再做A/B实验决定是否按类目分拆。

## 九、关键决策记录

| 决策 | 结论 |
|---|---|
| 模块方案 | A：通用+专属，模块跨类目最大复用 |
| 模块配置 | mandatory（不可取消）+ recommended（默认勾选）+ optional（默认不勾选，doubao可触发） |
| 类目推荐 | 系统根据三级类目自动推荐默认模块组，运营可增减 |
| comparison | 降为准通用，映射表按类目配置 |
| ingredient 命名 | food用`ingredient`、美妆用`beauty_ingredient`、医药未来用`pharma_ingredient` |
| 美食酒水内部分组 | A组基准 / A'组（粮油，ingredient可选）/ B组（生鲜）/ C组（滋补），通过二级覆盖实现 |
| 粮油调味 | A'组（A组基准但ingredient默认不勾选），二级覆盖 |
| 美妆保养保健 | 例外组（美妆专属模块降为optional），二级覆盖 |
| 映射覆盖策略 | 三层查找链：三级精确 → 二级覆盖 → 一级默认 → __default__ |
| 一二级编码 | 人工命名、一旦确定不修改（飞轮数据的持久化key） |
| 扩展顺序 | 美食酒水 → 美妆洗护 → 居家等纯通用 → 母婴亲子 → 数码家电 → 鞋包服饰 → 生鲜果蔬 |
| 数据飞轮 | 模块信号按类目编码隔离，风格Elo跨类目共享（≥3个一级类目后加观察指标） |
| 语料检索 | 三级 → 二级回退 → 一级回退，带confidence标记 |
| 向后兼容 | subCategory保留标记deprecated，catCode为新主键，flywheel_scores读新旧双key |
| 专属模块定义 | Phase 1-3写入代码类型，Phase 4+只写文档TODO |
