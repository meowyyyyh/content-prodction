# 多模态聊天优化方案：解决标记/JSON 泄漏问题

> 版本：1.0 · 日期：2026-07-22 · 状态：待评审

---

## 1. 问题根因

多模态聊天方案中，AI 输出混合了两种内容：

| 类型 | 格式 | 用途 |
|------|------|------|
| 自然语言 | 文案文本 | 给用户看的 |
| 机器指令 | `[IMAGE:N]` + `---LAYOUT---` JSON | 给前端解析的 |

两者在同一个文本流中混排。AI 每次采样都可能：
- 忘了写 `---LAYOUT---` 前缀 → JSON 裸露在文案中
- `[IMAGE:N]` 写成 `[IMAGE：N]`（全角冒号）
- JSON 中嵌套自然语言（"把图0放到 taste"）

这是"带内信令"的固有问题——**无法 100% 消除，只能降低概率或换架构**。

---

## 2. 方案对比

### 方案 A：结构化解耦（推荐）

AI 输出改为结构化的 JSON，文本和指令完全分离。

**前提**：Doubao-Seed-2.0-mini 的 Volcengine Ark 端点支持以下任一特性：
- `response_format: { type: "json_object" }`（强制 JSON 输出）
- Tool Calling / Function Calling

**实现**：

```
请求增加 response_format:
{
  "model": "ep-20260721203600-pz8qc",
  "response_format": { "type": "json_object" },
  "messages": [...]
}

AI 输出（纯 JSON，不进文案）：
{
  "modules": {
    "hook": "首屏钩子文案... [IMAGE:0] ... [IMAGE:1]",
    "taste": "口感体验文案..."
  },
  "layout": {
    "0": {"module": "taste"},
    "1": {"module": "taste", "group": "0"}
  }
}
```

前端解析：
```javascript
const data = JSON.parse(response)
const text = data.modules[key]  // 文案（含 [IMAGE:N]）
const layout = data.layout      // 排版指令
```

**优点**：
- JSON 永远不会泄漏到文案
- 100% 可解析，无格式变异
- `[IMAGE:N]` 仍在文本中保留位置信息

**风险**：
- Doubao 端点可能不支持 JSON Mode
- 强制 JSON 输出可能导致文案质量下降

### 方案 B：简化信令

砍掉 `---LAYOUT---` JSON，只保留 `[IMAGE:N]` 内联标记。

**实现**：

System prompt 简化为：

```
## 图片引用规则
1. [IMAGE:N] 是图片占位符（N从0开始）
2. 在文案中自然穿插 [IMAGE:N]
3. "图片放顶部" → [IMAGE:N] 放在文案最前面
4. "图片放底部" → [IMAGE:N] 放在文案最后面
5. "图文穿插" → [IMAGE:N] 分散在相关段落旁
```

前端只需要 `restoreImages()` 替换 `[IMAGE:N]`，不需要 JSON 解析。

**优点**：
- 零标记泄漏——`[IMAGE:0]` 文案中出现就是图片位置
- 删掉 ~50 行 JSON 解析代码
- AI 对简单标记的理解更稳定

**缺点**：
- 跨模块图片迁移需要用户手动拖拽（不能靠聊天指令）
- 复杂排版（分组、并列）表达能力不如 JSON

### 方案 C：混合容错

保持现状，但加强容错和解析。

**实现**：
1. `parseLayout` 同时匹配 `---LAYOUT---` 前缀和裸 JSON
2. 文案末尾的 JSON 清洗掉（不渲染）
3. 加更多 prompt 约束
4. 检测 `[IMAGE：N]`（全角冒号）并自动修正

**优点**：不改架构
**缺点**：治标不治本，问题仍会反复出现

---

## 3. 建议路径

```
Step 1: 验证 Doubao 是否支持 JSON Mode
         ↓ 支持                    ↓ 不支持
    方案 A（结构化解耦）        方案 B（简化信令）
         ↓                          ↓
    前后端改动 ~1天            前端简化 ~0.5天
```

### 验证 JSON Mode

用现有端点发一个测试请求：

```javascript
{
  "model": "ep-20260721203600-pz8qc",
  "response_format": { "type": "json_object" },
  "messages": [{
    "role": "system",
    "content": "你是JSON输出助手。输出JSON格式。"
  }, {
    "role": "user",
    "content": "返回 {\"text\":\"hello\",\"count\":1}"
  }]
}
```

如果返回正确的 JSON → 方案 A 可行。如果报错或不支持 → 方案 B。

---

## 4. 推荐

**先验证，5 分钟成本。支持就走方案 A，不支持就走方案 B。**

两个方案都能彻底解决 JSON 泄漏问题，区别只在于是否保留跨模块迁移的聊天指令能力。
