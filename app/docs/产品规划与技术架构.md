## 一、产品演进：3 个阶段，3 个产品形态

```
第一阶段                  第二阶段                  第三阶段
┌─────────────┐    ┌─────────────────┐    ┌───────────────────┐
│  AI 文案助手  │ → │  图文智能工作台   │ → │  用户私人内容工厂   │
│  帮你写文案   │    │  帮你写 + 配图    │    │  全链自动化         │
└─────────────┘    └─────────────────┘    └───────────────────┘
```

### 1.0 AI 文案助手（已完成 ✅）

输入商品信息 → AI 生成笔记结构文案 → 三版本并排对比：
  V1 = 默认风格（固定位置，初始小红书，随时间进化为个人复合风格）
  V2 = Elo 排名第 1 的纯风格（初始：日常闺蜜风）
  V3 = Elo 排名第 2 的纯风格（初始：趣味风）
→ 采纳/点踩 → 编辑定稿 → 导出。V1 越用越像"你"，V2/V3 按 Elo 动态排序。

### 2.0 图文智能工作台（当前阶段）

```
运营上传图片 → doubao vision API 自动分析
                  ↓
            输出：type（图片类型）+ layout_role（布局角色）
                + desc（详细描述）+ imageContentSummary（内容摘要）
                + imageOcrText（完整OCR文字）+ suggestedModule（建议模块）
                  ↓
            运营确认/修正 → AI 生成（参考图片分析结果）
                  ↓
            图文混排（单列全宽长图）→ 运营修改 → 导出
                  ↓
            导出自动存入预审语料库（corpus-review/）
```

**MVP 范围**：调制乳/风味牛奶一个品类簇（实际已扩展到乳制品全类目）。

**笔记格式**：单列全宽长图，图片顺序排列，不存在多图并排/网格布局。

**图片分析**：使用 doubao-seed-2.0-lite 视觉模型（doubao vision API），每张图独立调用，输出 type/desc/layout_role/imageContentSummary/imageOcrText/suggestedModule 六个字段。分析结果注入 Prompt 的"图片信息摘要"段落，AI 自然融入不直接拼接，保证文案和图片信息一致且避免图文冗余。

**> 已替代原方案中的 Tesseract.js OCR + 手动打标。doubao vision 同时完成了图片分类、OCR文字提取、布局角色判断和内容摘要，效果优于浏览器端 OCR。**

### 3.0 用户私人内容工厂

- 多平台适配：一套素材适配小红书/快团团/朋友圈
- 商品智能录入：拍照、扫码、粘贴链接自动提取
- 数据闭环：V1 默认风格从运营行为中自学进化（模块偏好 + L2 画像），V2/V3 由 Elo 排名驱动，L3 语料库全局共享（飞轮 L1 → L2 → L3 逐步开启）
- 用户知识库：历史笔记、产品偏好、客户画像

## 二、整体架构

```
                      用户浏览器
                 https://kuaigao.xxx.com
                          │
                          ▼
                 Cloudflare Pages
                  静态前端托管
                          │ /api/*
                          ▼
                 Render / Railway
                 Express 后端
                          │
        ┌────────┬────────┼────────┬────────┐
        ▼        ▼        ▼        ▼        ▼
    DeepSeek  doubao    语料库   合规扫描  用户画像
    (文本)   vision    (RAG)   (关键词)  (localStorage/JSONL)
              (图片分析)
                          │
                   ┌──────┴──────┐
                   ▼              ▼
              语料因子分析      编排器
              (analyzeCorpus)  (阶段 2+)
```

### 核心模块

**图片处理层**：上传 → doubao vision API 自动分析（type/desc/layout_role/imageContentSummary/imageOcrText/suggestedModule）→ 运营在确认弹窗中确认/修正 → 参与生成。

**> 2026-07 更新：已用 doubao vision API 替代原方案的 Tesseract.js + 手动打标。doubao 一次性输出图片分类、OCR文字、布局角色、内容摘要和建议模块，效果显著优于浏览器端OCR。**

**语料因子分析**：从语料库中提取品类级写作参数（字数、emoji密度、段落结构、开头方式等），注入V1默认风格的prompt。语料积累越多，参数越精确。

```
乳制品 → taste模块平均250字、emoji密度3.5%、闺蜜风为主
生鲜   → （待积累语料后提取）
零食   → （待积累语料后提取）
```

**编排器（Orchestrator，阶段 2+）**：

```
用户：帮我写一篇调制乳笔记，图片已上传

① 分析输入 → 品类识别 + 图片数据
② 检索参考 → 语料库找到同类高分笔记
③ 决策     → 选模板 + 图片分配到模块
④ 生成     → 组装 prompt → DeepSeek → 14 模块
⑤ 自检     → 合规 + 字数 + 标题污染 → 自动修复
⑥ 输出     → 图文混排结果
```

## 三、Agent 演进路径

当前本质是一个 UI + API 调用器 + 规则匹配引擎。核心洞察：**Agent 能力全部用规则引擎实现，不依赖模型升级。**

### 阶段 1：多工具（当前 — POC）

| 能力 | 方式 | 状态 |
|------|------|:--:|
| 文本生成 | DeepSeek | ✅ |
| 文件解析 | Word/Excel | ✅ |
| 图片打标 | 手动选择类型 | 🔜 |
| OCR 提取 | Tesseract.js | 🔜 |
| 模板匹配 | 品类→模板规则 | 🔜 |
| 语料检索 | RAG | 🔜 |
| 合规扫描 | 关键词匹配 | ✅ |

### 阶段 2：自主决策（规则引擎，3-5 天）

| 决策点 | 现在（人决定） | Agent（自己决定） |
|------|------|------|
| 模板 | 运营手动选 | 品类自动匹配 |
| OCR 文字 | 自动提取 | 自动注入 prompt |
| 图片位置 | 规则自动匹配 | 规则自动匹配 |
| 是否检索语料 | 人不说就不做 | 自动检索同类目 |

### 阶段 3：记忆 + 学习（统计，Phase 1 启动）

```
运营每次采纳小红书但改写为高端风 → 导出
    ↓
系统记录：高端风 Elo +4，小红书 Elo +1
    ↓
重复多次后高端风 Elo 超过默认三版末位
    ↓
下次生成自动替换：高端风进入默认三版
    ↓
运营不用手动调风格偏好——系统自己学会了
```

风格偏好跨类目共享（乳品学到的也用于零食），模块级不做自动干预——文字如何写得更好，是 PM 看数据后手动调 prompt。

### 阶段 4：自检 + 自修（规则引擎，2-3 天）

```
生成完 → Agent 自己跑一遍：
□ 合规扫描：有违禁词吗？→ 自动替换
□ 字数检查：达标了吗？→ 少 15 字 → 补 15 字
□ 标题检查：首行有污染吗？→ 自动删掉
□ 图片检查：有图没插入吗？→ 按模板塞进去
全部通过 → 输出
```

## 四、技术决策

### 图片处理：doubao vision API（已替代原Tesseract.js方案）

| 能力 | 方案 | 成本 |
|------|------|:--:|
| 图片内容识别 | doubao-seed-2.0-lite 视觉模型 | ~0.001元/张 |
| 图片类型判断 | doubao 自动分类 | 同上 |
| OCR文字提取 | doubao imageOcrText | 同上 |
| 布局角色判断 | doubao layout_role | 同上 |
| 内容摘要 | doubao imageContentSummary | 同上 |
| 运营确认 | ImageConfirmDialog（前端确认弹窗） | 免费 |

**> 原方案为 Tesseract.js（浏览器端OCR）+ 运营手动打标。2026-07已升级为 doubao vision API 全维度分析。**

### 语料：文本 + 视觉双层

| 语料类型 | 内容 | 用途 | 收集者 |
|----------|------|------|:--:|
| 文本语料 | 14 模块文案 | AI 参考写作风格 | 运营 |
| 视觉语料 | 品类→模板匹配表 | 图文排版 | 研发 |

**为什么文本和图片要放在一起？** 分开存只知道"有哪些文案"和"有哪些图片"，不知道配对关系。完整笔记级 JSON 才能知道首屏钩子配封面图、口感体验配产品图。

**入库方式**：

```
完整笔记 JSON
    │
    ├─ 文案 → 分模块切片 → 向量化 → RAG 检索
    └─ 完整笔记 → 存入语料库目录，作为图文参考
```

### 部署

| 层 | 平台 | 费用 |
|------|------|:--:|
| 前端 | Cloudflare Pages | 免费 |
| 后端 | Render | 免费层 |
| AI | DeepSeek API | 按量 |
| OCR | Tesseract.js（浏览器端） | 免费 |

## 七、竞争壁垒（护城河）

### 核心判断

通用技术没有壁垒——DeepSeek API 人人能用，前端可以抄，Prompt 可以逆向。**真正的壁垒不在代码里，在数据和使用习惯里。**

### 四层护城河

```
                    ┌─────────────────────────────┐
                    │  租户数据飞轮（最强）          │
                    │  ┌───────────────────────┐  │
                    │  │  用户习惯 + 记忆       │  │
                    │  │  ┌─────────────────┐  │  │
                    │  │  │  品类知识体系    │  │  │
                    │  │  │  ┌───────────┐  │  │  │
                    │  │  │  │ 通用技术   │  │  │  │
                    │  │  │  └───────────┘  │  │  │
                    │  │  └─────────────────┘  │  │
                    │  └───────────────────────┘  │
                    └─────────────────────────────┘
```

| 护城河 | 什么时候成形 | 强度 | 竞争对手要花多久 |
|------|:--:|:--:|:--:|
| 通用技术 | 现在 | ⭐ | 2 周 |
| 品类知识体系 | 2.0 开始积累 | ⭐⭐⭐ | 需要真实业务场景打磨 |
| 用户习惯 + 记忆 | 3.0 | ⭐⭐⭐⭐ | 3-6 个月（运营需深度使用） |
| **租户数据飞轮** | 2.0 启动 | ⭐⭐⭐⭐⭐ | **无法复制** |

### 第一层：品类知识体系

不是"模板比别人多"，而是对垂直品类的 know-how：

```
乳制品笔记的隐含知识：
  - 酸奶 taste 用三幕结构（前中后调），≥240 字
  - 纯牛奶 trust 要突出蛋白质含量
  - 调制乳 hook 要强调风味独特性
  - 品牌图放在信任模块前面效果好
```

这些规则散落在语料 JSON、模板定义、Prompt 调优记录和运营反馈里。**是运营和研发一起跑出来的，通用工具做不到。**

### 第二层：用户习惯 + LLM 画像分析

#### 核心思路

不是让运营告诉 AI 他喜欢什么风格，而是 **AI 自己从编辑行为中看出来**。参考豆包/Gemini 的记忆系统设计——用 DeepSeek 分析 DeepSeek 的输出，同一套 API 既能生成也能分析。

#### 工作流

```
运营采纳 → 中栏编辑 → 导出
              │
              ▼（后台异步，不阻塞）
┌─────────────────────────────────────────────┐
│  发送给 DeepSeek（一次轻量 API 调用）：        │
│                                               │
│  "这是 AI 生成的文案和运营修改后的终稿，        │
│   对比分析运营的写作偏好。提取：                │
│   1. 字数习惯（改长了还是改短了）              │
│   2. 句式特征（长句/短句、分段密度）           │
│   3. 用词偏好（喜欢/讨厌哪些词）               │
│   4. 结构偏好（加了什么模块、删了什么）         │
│   5. 语气倾向（正式/口语、热情/克制）          │
│   返回纯 JSON。"                               │
│                                               │
│  返回：                                       │
│  {                                            │
│    "sentence_style": "短句为主，每句≤25字",     │
│    "tone": "克制客观，不用感叹号",              │
│    "avoid_words": ["姐妹们","绝了","神仙"],     │
│    "prefer_words": ["喝喝看","蛮","还行"],      │
│    "module_tendency": {"taste":"+30%字数"},    │
│    "structure": "不要编号，不分段小标题"        │
│  }                                            │
└─────────────────────────────────────────────┘
    │
    ▼
存入用户画像 JSON → 下次生成时注入 prompt（只注参数不注内容）
    │
    ▼
画像增量加权合并（5:1 旧权重 vs 新分析，不因一次异常偏移）
```

#### 注入方式：只注参数，不注内容

**关键安全设计**——用户画像注入的是"怎么写"的抽象描述，不是"写了什么"的原文。避免跨品类内容污染：

```
❌ 危险：把历史终稿原文塞进去
   "以下是运营之前写的笔记：[320字全文]...请模仿这个风格"
   → 品牌名泄漏、内容串味

✅ 安全：只注入 LLM 提取的抽象特征
   "用户偏好：短句为主、不用'姐妹们'、每段2句以内、taste模块写长"
   → 纯风格描述，不涉及任何具体商品内容
```

#### 跨风格优先级

风格 > 用户画像。用户画像微调但不推翻全局风格：

```
小红书风（emoji 高频） + 用户画像（不用 emoji）
  → 风格优先，只在非核心区域微调（如字数、句式）
  
闺蜜风（emoji 少） + 用户画像（不用 emoji）
  → 自然叠加，效果更明显
```

#### 和语料库的关系

| | 公共语料库 | 用户画像 |
|------|------|------|
| 存什么 | 14 模块文案 + 图片映射 | LLM 提取的结构化偏好参数 |
| 作用 | 提升内容质量（"怎么写像好笔记"） | 个性化偏好（"怎么写像你"） |
| 注入方式 | RAG 检索 → prompt 参考 | 参数覆写 → 调整生成要求 |
| 会冲突吗 | — | 不冲突，参数和风格各管各的 |
| 跨风格安全 | ✅ | ✅ |

#### 投入

| 组件 | 实现方式 | 代码量 |
|------|------|:--:|
| 编辑行为统计 | 前端规则引擎 | ~200 行 |
| LLM 画像分析 | 一个分析 prompt + JSON parse | ~100 行 |
| 增量加权合并 | 工具函数 | ~50 行 |
| prompt 参数注入 | 改 buildPrompt | ~30 行 |
| **总计** | **核心代码 < 400 行** | |

#### 用户感知

不弹窗说"我学会了XXX"，而是**悄悄变好用**：

| 使用次数 | 感知 |
|:--:|------|
| 第 1 次 | 和现在一样 |
| 第 5 次 | taste 模块字数比以前长了——更对胃口 |
| 第 10 次 | 一键生成的基本不用改 |
| 第 20 次 | 换个人用这台电脑，风格完全不对——系统学会的是"你" |

#### 和对话记忆的关系

两种系统独立运作，互不依赖：

| | 对话记忆（chatHistory） | 用户画像 |
|------|------|------|
| 目的 | 让聊天助手知道"刚才说了什么" | 让生成系统知道"你怎么写" |
| 记什么 | 最近 4 轮对话 | LLM 提取的抽象风格特征 |
| 触发方式 | 每次聊天自动 | 每次导出后异步分析 |
| 持久化 | 当前会话（刷新就丢） | localStorage（永久） |
| 状态 | ✅ 已实现 | 📋 3.0 规划 |

### 第三层：语料数据飞轮（已设计完整方案 ✅）

```
运营用工具生成笔记
    ↓
运营修改后发布 → 发布后的笔记就是新语料
    ↓
新语料入库 → RAG 检索质量提升
    ↓
下次生成更准 → 运营改得更少 → 更愿意用
    ↓（循环加速）
```

这个数据不存在于任何公开训练集。每个品类 5-8 篇不难，但 14 簇 × 8 篇 = 112 篇**图文精确配对的笔记级语料**，竞争对手做不到——因为他们没有运营在用。

**完整数据飞轮方案见：[数据飞轮建设方案.md](数据飞轮建设方案.md)**。核心设计要点：

- **V1 默认风格**：固定位置、动态进化——每个模块独立学习来源风格（从哪个版本采纳最多）+ L2 用户画像参数注入，越来越像"你"
- **V2/V3 Elo 排序**：6 种纯风格按 Elo 排名取 top 2，V1 不参与竞争
- **快捷按钮改写入 Elo**：采纳 V2 但改写为高端风 → 高端风 Elo 额外加分 → 累积后自动替换 V2/V3
- **模块不做自动干预**：不自动缩字/跳过/取消勾选，怎么改由 PM 看数据后手动调 prompt
- **三层飞轮**：L1 秒级（Elo 即时调权）、L2 分钟级（编辑 diff → LLM 提取用户画像）、L3 天/周级（终稿预审入库 RAG）
- **中栏编辑距离 = 质量真分数**：采纳了但改了一大半 ≠ 满意，编辑率过滤"假采纳"
- **分阶段启动**：MVP 期间只采集信号（铺管道不开阀门），验收后逐步开启 L1→L2→L3

### 第四层：多租户个性化（未来最强壁垒）

当接入更多品牌商和个人团长后，6 套通用风格不够用。每个租户需要自己的风格画像：

```
全局（6 套通用风格）
  │
  ├─ 认养一头牛
  │   ├── 专属开头："云贵山野入瓶"
  │   ├── 必提关键词：A2β-酪蛋白、9 种益生菌、GAP 认证
  │   ├── 话术偏好：不写"姐妹们"，用"喜欢酸奶的朋友"
  │   └── 历史语料：持续增长
  │
  ├─ 团长阿琳
  │   ├── 专属开场白："阿琳帮你试过了"
  │   ├── 结尾："信我的准没错"
  │   ├── 语气：闺蜜风 + 自己的口头禅
  │   └── 历史语料：持续增长
  │
  └─ ...

租户数据飞轮：品牌/团长用得越久 → 数据越多 → 越懂他 → 越离不开
```

**这一层的不可复制性最强**：不是"AI 比别人好"，而是"AI 更懂你的品牌"。品牌数据和风格画像无法导出给竞品，离开工具就没了。这是**双向锁定**：品牌离不开工具（数据在你这），工具也离不开品牌（他们是差异化来源）。

## 八、后续接入多租户的思考

### 品牌商 vs 个人团长，需求不同

| | 品牌商 | 个人团长 |
|------|------|------|
| 调性要求 | 高，品牌一致性严格 | 中，个人风格灵活 |
| 内容量 | 大，日均多篇 | 小，一周几篇 |
| 付费意愿 | 强（SaaS 订阅） | 中（按量或免费） |
| 定制深度 | 需要专属模型调优 | 需要快捷模板 |
| 护城河价值 | 极高（品牌数据不可迁移） | 中（个人数据量小） |

### 潜在演进路径

```
当前（单租户通用） 
  → 多租户风格画像（每个租户独立配置） 
  → 租户微调模型（品牌专属 Prompt / LoRA） 
  → 平台效应（品牌和团长互相引流）
```

优先做品牌商——数据量大、付费意愿强、护城河最牢。
