# 语料动态评分系统设计方案 v2

> 2026-07-20 · v2 合入 Codex 审阅反馈（15条） · 替代静态1-5星打分的动态评分机制
> 原则：系统自动算，不用人打分，幕后黑盒

---

## 一、核心思路

不用人打分。每次搜索语料时，系统根据三个维度实时计算每篇语料的得分，选出最适合当前场景的那篇。

```
score = 质量分 × 验证分 × 时效分
```

三个维度各管各的。**质量是基础，验证是放大器，低质量语料验证再多也无法翻身。**

---

## 二、评分公式

### 2.1 质量分

```
质量分 = (1 - 编辑率) × 原创性系数

编辑率 = (定稿字数 - 快照字数) / 快照字数   （范围 0~1）

原创性系数：
  moduleOrder ≠ DEFAULT_MODULE_ORDER  → 1.0    （用户动手调过顺序）
  moduleOrder ＝ DEFAULT_MODULE_ORDER  → 0.7    （没调顺序，信息量低）
```

| 编辑率 | 编辑率分 | × 原创(1.0) | × 原创(0.7) |
|:--:|:--:|:--:|:--:|
| 10% | 0.90 | **0.90** | 0.63 |
| 30% | 0.70 | **0.70** | 0.49 |
| 50% | 0.50 | 0.50 | 0.35 |
| 70%（入库上限） | 0.30 | 0.30 | 0.21 |

> **已知取舍**：原创性系数是二元判断（调了/没调）。运营可能有调整顺序但导出时看不出（满意内容就直接导出了），这种情况会被系统性判为0.7——实际上只是不关心顺序，不是调得不好。未来可升级为连续值（编辑距离），Phase 1 先二元。

### 2.2 验证分

```
验证分 = 1 + ln(1 + 引用计数)

引用计数 = 整数（不是浮点），由以下事件累加：

  采纳整个版本（"采纳此版本"）      → +moduleCount（即该语料 moduleOrder.length）
  采纳单个模块（"采纳"按钮）        → +1
  修改后导出                      → +moduleCount / 2（单独导出会话）
```

| 事件 | 叠加规则 |
|------|------|
| 同一导出会话中，先采纳版本又修改导出 | **取最大值**（+moduleCount 和 +moduleCount/2 不叠加），因为这是同一次行为的两个观测信号 |
| 不同导出会话 | 独立计算 |

**冷启动保护**：引用计数未达到 moduleCount 时不启动验证分提升（即验证分保持 1.0），至少要有一次完整版本采纳或等量模块采纳才开始增长。

| 引用计数 | 验证分 = 1+ln(1+count) | 说明 |
|:--:|:--:|------|
| 0 | 1.00 | 从没被参考过 |
| 11（≈1次版本采纳） | 1+ln(12) = **2.48** | 这是为 moduleCount=11 时一次完整采纳对应的计数（11×1）。注意这里的 count 是11不是1。 |

> **修正说明**：重新看上面的例子——如果一次版本采纳 = +moduleCount（比如11），引用计数直接到11，验证分跳到 1+ln(12)=2.48。这意味着首次版本采纳后验证分就从1.0跳到2.48，这看起来不合理。应该让版本采纳 +moduleCount 等价于采纳了所有模块一次——但"第一次被采纳"和"被采纳了11次"在心理上应该有区分。

> **结论**：保持原梯度设计但改用整数计数 + 对数映射。单模块采纳 +1，全版本采纳 +moduleCount（等价采纳该语料的每一个模块），修改导出 +moduleCount/2。对数映射确保早期信号有价值但后期不会爆炸。

**重算示例**（moduleCount=11）：

| 事件序列 | 累计引用计数 | 验证分 = 1+ln(1+count) |
|------|:--:|:--:|
| 无引用 | 0 | 1.00 |
| 采纳了3个模块 | 3 | 1+ln(4)=2.39 |
| 采纳整个版本 | 11 | 1+ln(12)=3.48 |
| 版本采纳 + 5个模块被单独采纳 | 16 | 1+ln(17)=3.83 |
| 大量使用（约3次版本采纳） | 33 | 1+ln(34)=4.53 |
| 超高频使用（10次版本采纳） | 110 | 1+ln(111)=5.71 |

**对数映射效果**：从0到1次版本采纳（0→11）验证分从1.0跳到3.48——增长很快；从3次到10次（33→110）只从4.53到5.71——增长放缓。赢了也不会通吃。

### 2.3 时效分

```
渐进衰减（替代 v1 的 25 天一刀切到 0.50）：

  0-7天     → 1.00 - 距今天数 × 0.03    （7天→0.79）
  8-25天    → 0.79 - (距今天数-7) × 0.015 （25天→0.52）
  26-60天   → 0.52 - (距今天数-25) × 0.006 （60天→0.31）
  60天后    → 0.31 - (距今天数-60) × 0.002 （最低 0.15）
```

| 距今天数 | 时效分 |
|:--:|:--:|
| 今天 | 1.00 |
| 7天 | 0.79 |
| 25天 | 0.52 |
| 60天 | 0.31 |
| 120天 | 0.19 |
| 150天+ | 0.15（触底） |

**为什么分四段**：避免了 v1 中所有 25 天+ 语料挤在 0.50 一个分数档的问题。25天和60天的语料现在有区分度。旧语料靠验证分仍可竞争但需要更强的验证数据支撑。

## 三、完整计算示例

同品类+同风格下 3 篇语料（moduleCount=11）：

| 语料 | 编辑率 | 原创 | 引用计数 | 天数 | 质量分 | 验证分 | 时效分 | **总分** |
|------|:--:|:--:|:--:|:--:|------|------|------|------|
| A | 10% | ✅ 1.0 | 16 | 3 | 0.90 | 3.83 | 0.91 | **3.14** |
| B | 50% | ✅ 1.0 | 0 | 30 | 0.50 | 1.00 | 0.50 | **0.25** |
| C | 20% | ❌ 0.7 | 33 | 14 | 0.56 | 4.53 | 0.69 | **1.75** |

A 质量最高+较新鲜 → 排第一。C 被大量引用 → 排第二（验证分拉上来了，虽然时效和质量一般）。B 质量一般+无引用+旧 → 排第三（0.25，永远进不了 Top-3，但数据飞轮方案里编辑率>70%已经不入库了，B 本就不该在池子里）。

## 四、选择策略

### 4.1 基本策略：Top-3 加权随机

取总分最高的 3 篇 → 按分数占比分配概率：

```
语料A总3.14 → 占比 61% → 中奖概率 61%
语料C总1.75 → 占比 34% → 中奖概率 34%
语料B总0.25 → 占比  5% → 中奖概率  5%
```

### 4.2 探索概率（分层）

```
90% → Top-3 加权随机（如上）
 5% → 完全随机（所有候选均等概率）
 5% → 从 0 引用语料中随机选（给未被验证的语料被发现的通道）
```

> 候选 ≤ 3 篇时关闭全部探索概率（池子太小，没有"外面"可选）。

### 4.3 新语料加温

首次入库的语料，在前 5 次搜索中验证分临时 +0.5。有了真实引用后加温自动失效（引用计数 > 0 即失效）。

作用：新语料入库时引用计数为 0、验证分 1.0，如果没有高编辑率支撑，根本挤不进 Top-3。临时 +0.5 让新语料有机会被看到，被采纳后就有真实引用，加温也随之失效。

### 4.4 V2 和 V3 如何分配

V2 和 V3 各选**不同**语料：

```
V2：加权随机选一篇（比如选中了A）
V3：从剩余候选里再加权随机选一篇（排除A）
```

边界情况：

| 候选数 | 行为 |
|:--:|------|
| ≥ 2 | V2 选中A → V3 从剩余选B，正常 |
| = 1 | V2 用这篇 → V3 打乱 DEFAULT_MODULE_ORDER（随机交换 2 个模块位置），避免 V3 和 V1 出现相同顺序 |
| = 0 | 都兜底 DEFAULT_MODULE_ORDER，V3 打乱 |

## 五、和 Elo 的关系

两条线独立运作，数据共享：

```
语料动态评分 → 决定"参考哪篇语料的模块顺序"（品类+风格级）
Elo 风格排名 → 决定"V2/V3 用哪个风格"（用户+风格级）
```

**隐含的反馈循环**（已知设计特性，非 bug）：

- 高 Elo 风格的语料 → 更频繁被 V2/V3 引用 → 引用计数涨得快
- 低 Elo 风格的语料 → 即使质量更好，用户不选该风格 → 引用计数涨得慢

这不是系统缺陷——引用计数本身反映的就是"实际使用频次"。高质量但风格不受欢迎的语料引用少是合理的（选风格是用户偏好的表达，语料评分只能决定"同风格下谁更好"，不能决定"用户该选哪个风格"）。

数据飞轮方案中的 10% 探索概率（偶尔换掉 V2/V3 中的某一个风格）可以在长期缓解这个反馈循环。

## 六、数据存储

### 6.1 语料 JSON 新增字段

```json
{
  "editRate": 0.15,
  "moduleOrderOriginal": true,
  "referenceCount": 16,
  "convertedAt": "2026-07-18T00:00:00.000Z"
}
```

| 字段 | 类型 | 写入时机 | 说明 |
|------|------|------|------|
| `editRate` | number | 导出时 | 编辑率 0~1。null 时默认 0.3（中性回退） |
| `source` | string | 入库时 | 语料来源。`"curated"` = 人工经入库评分卡筛选入库；`"auto"` = L3 飞轮自动入库。详见《种子语料入库评分体系设计方案》 |
| `seedScore` | number | 入库时 | 入库评分卡总分（0~5）。curated 语料有此字段，作为动态评分初始种子值；auto 语料此字段为 null |
| `moduleOrderOriginal` | boolean | 导出时 | null 时默认 false（保守，取 0.7 系数） |
| `referenceCount` | integer | 采纳时递增（API 调用） | 整数，非浮点。搜索 API 计算验证分用 |
| `convertedAt` | string | 导出时（已有字段） | 时效分计算基准 |

### 6.2 新增 API

| 端点 | 说明 |
|------|------|
| `POST /api/corpus/reference` | 采纳时调用，递增 referenceCount。body: `{productName, catLevel1/2/3, delta: integer}` |
| `GET /api/corpus/search` | 搜索语料，对每篇匹配语料实时计算动态总分，按总分降序返回 top N。**返回的排序依据是实时动态分（不是 JSON 中的 corpusScore 字段）** |

### 6.3 API 写并发保护

`POST /api/corpus/reference` 使用 append-only 模式：

```
data/corpus-review/
└── reference_log/
    └── {date}.jsonl    ← 追加写入，每行一条引用事件

格式：{"productName":"...","catLevel1":"...","catLevel2":"...","catLevel3":"...","delta":1,"timestamp":...}
```

搜索时读取所有 log 文件的引用事件，按 productName+catCode 聚合得到每篇语料的 referenceCount。避免多运营同时采纳时的 JSON 文件覆盖写入问题。

### 6.4 字段命名统一

| 位置 | 字段名 | 说明 |
|------|------|------|
| 语料 JSON | `referenceCount` | 存储引用次数（整数） |
| 搜索 API 返回 | `score` | 实时计算的动态分（质量×验证×时效），不是 JSON 中的静态字段 |
| 语料 JSON（保留） | `corpusScore` | 运营手动评分预留字段，当前始终为 null，暂不使用 |

## 七、前端需要记录的数据

### 生成时：记住每版参考了哪篇语料

```typescript
const versionCorpusRef = useRef<Record<string, {productName: string; catCode: string}>>({})
// { 'v2': {productName:'认养一头牛...', catCode:'美食酒水::酒水饮料::乳制品'}, 'v3': {...} }
```

### 采纳时：更新该语料的引用计数

```
用户采纳 V2 的 taste 模块
  → 读 versionCorpusRef['v2'] → 找到语料标识
  → POST /api/corpus/reference { productName, catLevel1/2/3, delta: 1 }
  → 后端 append 到 reference_log/{date}.jsonl

用户点击"采纳此版本" V2
  → POST /api/corpus/reference { ..., delta: moduleCount }
  → moduleCount = 该语料 moduleOrder.length（动态读取）

用户修改后导出（单独会话，没有先采纳版本）
  → POST /api/corpus/reference { ..., delta: moduleCount/2 }
```

## 八、冷启动行为

| 场景 | 行为 |
|------|------|
| 语料 ≤ 3 篇 | 全部返回，加权随机。关闭探索概率（无"外面"可选） |
| 语料 = 0 篇 | 返回空，前端兜底 DEFAULT_MODULE_ORDER |
| 语料都没有引用记录 | 质量分+时效分排序（验证分全是1.0） |
| 语料缺少新字段（editRate/moduleOrderOriginal 为 null） | editRate→默认0.3，moduleOrderOriginal→默认false（保守） |
| 语料引用计数未达 moduleCount | 验证分保持 1.0（冷启动保护，至少一次完整采纳才启动） |
| 种子语料（经入库评分卡筛选，source=curated） | 入库评分卡总分映射为初始引用分（referenceCount = floor(seedScore × 3)），冷启动期优先引用 |
| 新入库语料（前5次搜索） | 验证分临时 +0.5，引用计数>0后失效 |
| 风格不匹配 | 三层品类回退（level3→level2→level1），最终兜底 DEFAULT_MODULE_ORDER |
| 候选=1 且 V3 兜底 | V3 打乱 DEFAULT_MODULE_ORDER（随机交换2个模块） |
