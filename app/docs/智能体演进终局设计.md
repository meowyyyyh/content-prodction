## 一、终局产品形态

### 1.1 界面结构

单栏对话式界面 + 底部工具栏 + 可展开的手动编辑抽屉：

```
┌───────────────────────────────────────┐

│                                       │
│  ┌─────────────────────────────────┐  │
```

### 1.2 对比当前三栏布局

| 当前（工具） | 终局（智能体） |
|------|------|
| 三栏并排：左面板 + 中栏 + 右面板 | 单栏对话流 + 可展开编辑抽屉 |
| 表单填写 → 点按钮 → 等生成 | 对话驱动：说需求 → Agent 做事 → 给结果 |
| 三版并排对比（V1/V2/V3） | 单版输出，对话中调整 |
| 运营在三个面板间切换注意力 | 运营只看一个对话流 |
| 编辑是独立的中栏 contentEditable | 对话自然语言改写为主，手动编辑抽屉为辅 |

### 1.3 核心交互模型

```
用户行为           →  Agent 做什么          →  输出

丢文件/图片         →  感知层：分析+提取      →  告知识别结果+缺口
说需求              →  决策层：确定品类/模块/风格 →  确认或直接执行
确认/调整           →  执行层：生成笔记       →  流式输出
说"改XX"           →  执行层：改写目标模块    →  流式输出
说"导出"           →  执行层：导出+学习采集   →  文件下载
```

### 1.4 与当前架构的本质差异

| 维度 | 工具（当前） | 智能体（终局） |
|------|------|------|
| 界面 | 三栏表单 + 预览 | 单栏对话流 + 编辑抽屉 |
| 交互 | 填表 → 点按钮 → 等待 | 说需求 → 确认 → 迭代修改 |
| 版本数 | 三版并排竞争 | 单版，对话中调整 |
| 风格选择 | 三版竞争 → Elo 排名 | Agent 根据 Elo 记忆默认值，不确定时主动询问 |
| 模块顺序 | 三版各自独立取序 | 单版融合学习+语料+用户偏好 |
| 学习反馈 | 采纳快照 + 导出学习 | 对话中的每一次交互都是学习信号 |
| 编辑方式 | contentEditable 富文本 | 对话自然语言改写为主，手动抽屉为兜底 |

## 二、两种交互模式

### 2.1 精编模式（默认）— 按模块组确认

**设计意图**：像和编辑协作——按模块组分批确认，每写完一组停下来等反馈。兼顾控制感和效率。适合新品类、高质量要求的笔记。

**模块组划分（以美食酒水 14 模块为例）**：

```
组1「开场+报价」：hook → price            （定基调，最关键）
组2「口感+品质」：taste → trust → ingredient → origin  （核心卖点）
组3「品牌+场景」：brand → scene            （感性内容）
组4「物流+贴士」：aftercare → tips          （实用信息，可跳过）
组5「收尾」    ：faq → cta                 （转化+答疑）
```

**非必选模块（feedback、comparison）**：如果运营勾选了则归入组 4 或组 5，未勾选则跳过。

**交互流程**：

```
Phase 1：素材分析
  用户: [丢文件/图片]
  Agent: 识别到XX商品，规格XX，售价XX。
         还缺：配料表、品牌故事。
         有的话丢给我，没有我先写着。
         风格用小红书？还是换个？

Phase 2：逐组生成 + 确认
  Agent: —— 组1「开场+报价」——
         【hook】
         姐妹们！！挖到宝了！！A2β酪蛋白酸奶...
         【price】
         💰 ¥59.90/200g×12瓶，单瓶不到5块...
         这组OK吗？[OK] [改hook] [改price] [换风格]

  用户: 行

  Agent: —— 组2「口感+品质」——
         【taste】... 【trust】... 【ingredient】... 【origin】...
         这组OK吗？

  用户: taste太短了，多写点感官细节

  Agent: [改写 taste，扩充到 250 字]
         这样呢？

  用户: 可以

  Agent: —— 组3「品牌+场景」—— ...
  ... 直到所有组写完

Phase 3：收尾
  Agent: 全部写完了。物流售后已贴在末尾。
         [完整笔记预览]
         要导出吗？还是再调哪里？
```

**特征**：
- 按模块组分批，5 组确认 vs 14 个模块逐个确认——效率显著提升
- 用户可对组内单个模块提修改（"taste太短了"），不强制整组通过
- 随时可以说"剩下的直接写" → 切到粗编模式
- 每次组确认和单模块修改反馈都是学习信号

**适用**：新品类、质量要求高、运营刚开始使用。

### 2.2 粗编模式（速写）— 仅关键模块停下

**设计意图**：像雇了个熟手写手——只在最关键的少数模块停下确认，其余全自动。适合成熟品类、批量出稿。

**关键模块（按品类可配置）**：

粗编模式下的关键模块不硬编码，而是按一级类目定义。"必停"模块的选择逻辑：**调性锚点 + 核心卖点 + 敏感信息 + 收尾转化**。

```typescript
// 粗编模式关键模块映射表（按一级类目配置）
const COARSE_KEY_MODULES: Record<string, ModuleKey[]> = {
  '美食酒水':   ['hook', 'taste', 'price', 'cta'],
  '美妆洗护':   ['hook', 'beauty_effect', 'price', 'cta'],
  '__default__': ['hook', 'price', 'trust', 'cta'],  // 通用类目
}
```

以美食酒水为例：其余 10 个模块（trust / ingredient / origin / brand / scene / aftercare / tips / feedback / comparison / faq）一次性输出，不停下确认。

**交互流程**：

```
Phase 1：素材分析（同精编模式）
  用户: [丢文件/图片]
  Agent: 识别到XX商品。缺配料表，先写着。

Phase 2：关键模块确认
  Agent: —— hook ——
         姐妹们！！挖到宝了！！...
         [OK] [换一个] [自己写]

  用户: 行

  Agent: —— taste ——
         打开就能闻到浓郁的奶香...
         [OK] [写长点] [换个风格]

  用户: 写长点

  Agent: [扩充 taste]
         这样呢？

  用户: 可以

  Agent: —— price —— ...
  Agent: —— cta —— ...

Phase 3：其余模块一次性输出
  Agent: 剩下的我直接写了——
         [trust + ingredient + origin + brand + scene + aftercare + tips + faq]
         全部在这里，要调哪里告诉我。

Phase 4：收尾
  用户: 导出
  Agent: [导出文件] 记住了：taste偏长、price简洁。
```

**特征**：
- 只在 4 个关键模块停下确认，其余全自动
- ~5-8 轮对话完成一篇笔记
- Agent 全流程自主决策品类→模块→风格→顺序
- 默认用历史学习到的偏好

**适用**：写过很多次的品类（学习系统已收敛）、批量出稿、运营已信任系统判断。

### 2.3 两种模式对比

| 维度 | 精编模式（默认） | 粗编模式（速写） |
|------|------|------|
| 确认策略 | 按模块组分批（5 组） | 仅关键模块（按品类 3-4 个）|
| 对话轮数 | ~10-15 轮 | ~5-8 轮 |
| Agent 决策权 | 中 — 分组确认 | 高 — 仅关键处停下 |
| 学习信号密度 | 高（每组确认 + 模块级反馈） | 中（关键模块反馈 + 不满修改） |
| 适用阶段 | 新品类、新用户 | 成熟品类、熟练用户 |
| 用户体验 | 掌控感强 | 效率高 |

### 2.4 重要区分：模式切换 vs 跳过确认

文档中"切到粗编模式"这个说法实际包含两个独立概念，实现时需区分：

| 概念 | 触发方式 | 行为 | 是否改变全局模式 |
|------|------|------|:--:|
| **模式切换** | 底部工具栏手动选择 | 变更整场对话的默认生成策略 | ✅ 是 |
| **跳过确认** | 精编中途说"剩下的直接写" | 在当前生成中跳过剩余确认步骤，一次性输出 | ❌ 否——下次新笔记仍用精编 |

```
示例：
  精编模式 → 组 2 完成后用户说"剩下的直接写"
    → 触发"跳过确认"：组 3/4/5 一次性输出（不是切到粗编模式）
    → 这篇笔记写完后，下次新笔记仍然默认精编模式，按组确认

  精编模式 → 用户手动在工具栏切换到粗编
    → 触发"模式切换"：从下一笔笔记开始直接用粗编策略
    → 全局默认值变更
```

澄清这个区分后，粗编→精编的转换也变清晰了：

```
粗编模式中途说"一个一个来"
  → 从下一个关键模块起改为分组确认
  → 需要"模块→组"映射表来定位当前属于哪个组
  → 例如粗编中刚完成 price（关键模块 #3），
     映射表查询：price 属于组 1「开场+报价」
     下一组应为组 2「口感+品质」
  → 从组 2 开始按精编流程走
```

**精编模式下单模块修改后的复现流程**：用户对组内某模块提修改（如"taste太短了"）→ Agent 改写该模块 → 单独展示确认 → 确认后重新呈现整组摘要 → 用户说"行"才进入下一组。

## 三、模式切换设计

### 3.1 切换入口

位置：对话区底部工具栏：

```
┌─────────────────────────────────────────────────┐
│ 📎 上传  ✏️ 手动编辑  📥 导出  [精编 ▾]  ⚙️     │
│                               ├ 精编（分组确认）  │
│                               └ 粗编（快速出稿）  │
└─────────────────────────────────────────────────┘
```

### 3.2 切换行为

| 场景 | 行为 | 是否持久化 |
|------|------|:--:|
| 会话开始前手动切换 | 整场对话用所选模式 | ✅ 更新默认偏好 |
| 精编中途说"剩下的直接写" | **跳过确认**（非模式切换）：剩余模块一次性输出。下次新笔记仍用精编 | ❌ 仅当前笔记 |
| 粗编中途说"一个一个来" | 从下一个关键模块起改为分组确认。借助模块→组映射表定位当前所属组 | ❌ 仅当前笔记 |
| 会话中途手动点击切换 | 从下一笔记起按新模式执行。当前正在生成的笔记不受影响 | ✅ 更新默认偏好 |

### 3.3 默认值策略

```
新用户（无历史数据）→ 默认精编模式
  ↓ 累计导出 ≥ 15 篇 且 编辑率 < 20%
  → Agent 主动建议："我看你写得挺顺的，要不要切粗编？"
  → 用户确认 → 切换
  → 用户拒绝 → 保持精编

老用户（多品类）→ 按品类区分
  新品类 → 精编模式
  写过 5 篇以上的品类 → 粗编模式
```

### 3.4 手动编辑抽屉

保留 contentEditable 富文本编辑器作为**可展开的底部抽屉**。使用场景：

- 运营想直接打字微调某段文案，不想走对话
- 对话改了好几轮还不满意，想自己动手
- 修改自动同步回对话上下文（Agent 知道用户手动改了什么）

抽屉平时收起，点击底部工具栏 ✏️ 按钮展开。展开时对话区缩小但不关闭——运营可以边看对话边手动改。

### 3.5 模式对 Agent 行为的影响

本质上是参数注入到 Agent 的 system prompt：

```
精编模式：
  "按模块组分批生成：组1(hook,price)→组2(taste,trust,ingredient,origin)→
   组3(brand,scene)→组4(aftercare,tips)→组5(faq,cta)。
   每写完一组停下来等用户确认。用户说'行'才继续下一组。
   用户可对组内单个模块提修改要求。用户说'剩下的直接写'就切到粗编。"

粗编模式：
  "一键生成完整笔记，仅在关键模块处停下确认（具体哪些模块取决于品类，
   见 COARSE_KEY_MODULES 映射表，美食酒水为 hook/taste/price/cta）。
   其余模块不停顿直接输出。用你记忆中的用户偏好直接决策。
   只在素材严重不足或完全新品类时才主动提问。"
```

### 4.2 意图路由设计

```typescript
type AgentIntent =
  | 'generate_note'      // 生成完整笔记
  | 'analyze_material'   // 分析素材（文件/图片）
  | 'rewrite_module'     // 改写指定模块
  | 'expand_module'      // 扩充指定模块
  | 'shorten_module'     // 缩短指定模块
  | 'change_style'       // 换风格重写
  | 'export_note'        // 导出
  | 'ask_question'       // Agent 在问用户问题（用户回复后的二次路由）
  | 'chat'               // 闲聊/问问题

interface IntentInput {
  userMessage: string
  attachedFiles: File[]
  currentState: AgentState
  conversationHistory: Message[]
}

interface IntentResult {
  intent: AgentIntent
  confidence: number
  params: Record<string, any>  // 提取的参数（模块名、风格名等）
}
```

意图路由使用一次轻量 LLM 调用，不依赖规则匹配——自然语言的灵活性（"taste 写长点" / "口感那块再加点" / "第二个模块扩充一下"）决定了规则覆盖不全。

### 4.5 为什么不需要复杂 Agent 框架

当前工具的每个能力都已经是独立、可测试的 API 端点。把它们挂到 Agent Loop 上只需要四样东西：

```
Agent Loop = 意图路由 + 工具调度 + 上下文组装 + 对话状态管理
              ↑           ↑           ↑              ↑
           一次 LLM    函数调用    现有 API 的     服务端 JSON
           (~200tokens) 无框架     输入拼接        文件
```

- **意图路由**：一次轻量 LLM 调用，不需要单独的分类模型
- **工具调度**：纯函数调用，没有外部依赖
- **上下文组装**：在现有 `buildPrompt` 的参数拼接基础上扩展，核心逻辑不变
- **对话状态管理**：服务端一个 JSON 文件记录当前 session 的 state + history

不需要 LangChain、MCP、或复杂的 tool-calling 框架。Agent 的 system prompt 中声明可用工具（标准 tool use 格式），LLM 返回工具名 + 参数，后端执行后把结果塞回对话上下文。**核心编排逻辑 < 300 行。**

#### 工具调用保障机制（实施时需实现）

1. **Tool-call guard**：LLM 返回的工具调用可能格式错误或引用不存在的工具。在工具调度层加 JSON 解析 + schema 校验。校验失败 → 最多重试 2 次（将字段级 error 报文重新提示 LLM 修正）。2 次后仍失败 → 降级为纯 LLM 回复（不调工具，用对话上下文直接回答），同时记录隐式错误信号到 `system_errors` 日志。

2. **Token 预算管理**：多次工具调用（analyze_images → check_gaps → ask_user → generate_module × 5）会在单次用户 turn 内消耗数千 tokens。对单次 Agent turn 设 8000 token 软上限，超限后压缩中间结果再进下一轮。

3. **长对话截断策略**：100+ 轮对话会爆 token。当 `conversationHistory` 累计超过 6000 tokens 时，保留最近 20 轮对话 + 商品信息摘要 + 用户画像摘要，早期对话做摘要压缩。

## 五、学习系统（单版本版）

### 5.1 单版本下学习维度的变化

相比三版本竞争模式，单版本对话式 Agent 的学习信号更丰富：

| 学习维度 | 三版本下 | 单版本下 | 变化 |
|------|------|------|:--:|
| 模块顺序 | V1 从采纳快照学 | 唯一版本从导出时的最终顺序学 | **简化**——不需要三版分开 |
| 风格偏好 | 三版竞争 → Elo 排名 | 用户选择 + Agent 记忆 → Elo 排名 | **算法不变**，消费方式变 |
| 文风句式 | L2 画像分析编辑 diff | 同上 + 对话中的自然语言反馈 | **信号更丰富**|
| 排版偏好 | 图片位置关系 | 不变 | 不变 |
| 编辑率 | 快照 vs 定稿 | Agent 生成内容 vs 用户修改后最终版 | **信号更精确**——对话中每轮修改都可算 delta |

### 5.2 对话中的新信号源

单版本对话式 Agent 比三版本工具多了丰富的隐式信号：

| 信号 | 来源 | 学习什么 |
|------|------|------|
| "行" / "可以" | 用户确认一个模块/模块组 | 该模块在当前风格下的生成质量 +1 |
| "太短了" | 用户不满意字数 | 该模块默认字数偏少 → 下次上调 |
| "别用姐妹们" | 用户否定措辞 | L2 画像：avoid_words 加入"姐妹们" |
| "换个高端风" | 用户切换风格 | 高端风 Elo + 当前风格 Elo - |
| "剩下的直接写" | 用户从精编切粗编 | 用户对系统判断的信任度上升 |
| "一个一个来" | 用户从粗编切精编 | 用户对系统不信任 / 新品类谨慎 |

三版本工具只有采纳/点踩/编辑率三个信号维度。对话式 Agent 中**每一次交互都是信号**，维度远比三版本丰富。

**补充信号（Codex 审阅建议）**：

| 信号 | 来源 | 学习什么 |
|------|------|------|
| 同模块连续修改 ≥ 3 次 | 同一模块/同一轮对话内反复要求改写 | 该模块在该风格×品类下的质量分 -0.5——Agent 对用户的预期存在系统偏差 |
| 对话中断（24h 未导出） | 会话状态后台监控 | 该会话所有 Agent 决策权重打折 × 0.5；连续 3 次同品类中断 → 该品类默认风格 -1 |
| 模块交界处编辑率高 | taste→price 之间的用户手动修改量 | 模块过渡偏好——当前 prompt 缺少模块间过渡引导，需调整上下文衔接 |

### 5.3 风格 Elo 的新定位（单版本适配）

**核心问题**：Elo 的数学基础是成对比较——两个版本正面交锋，赢者得分。三版本模式下 V2 和 V3 天然形成对局。单版本对话模式没有成对比较——用户对唯一版本的满意/不满意是绝对评价，不是相对评价。

**解决方案**：保留 Elo 算法本身（数据结构、K 因子、时间衰减、prompt 版本戳完全不变），但信号输入方式改为**导出频次模拟对局**：

```
三版本成对比较（阶段 1）：
  采纳 V2 → V2 战胜 V3 → V2 Elo +K, V3 Elo -K

单版本导出模拟（阶段 3）：
  每次导出时，导出所用风格 vs 该品类下其他已导出风格做一次模拟对局：
  · 用户在品类 X 下用 A 风格导出 7 次、B 风格导出 3 次
    → 导出时 A 战胜 B（频次高 = 偏好强）
  · 用户说"换个高端风" → 高端风战胜当前风格（虚拟对局，单次有效）
  · 快捷按钮"改写为XX风" + 导出 → XX 风额外加分（权重 ×2，改写意愿 > 被动接受）
```

**为什么模拟对局可行**：Elo 本质是"从比赛结果反推实力"。成对比较是理想信号源但不是唯一的——导出频次本质上也是"选择了 A 而非 B"的表达，只是信号粒度更粗（一次导出 vs 一次采纳）。数学基础没变，变的是信号的**采集方式**。

**消费方式**：
- ① 用户没指定风格时，Agent 默认用 Elo 最高分风格
- ② 用户说"换个风格"时，Agent 按 Elo 排名推荐备选（top 2）
- ③ 快捷按钮"改写为XX风"使用频率 → Elo 加分
- ④ 某风格 Elo 持续垫底 → Agent 不再主动推荐，但用户点名时仍可用

### 5.4 模块顺序学习的简化

三版本中 V1/V2/V3 各自独立取序。单版本下合并为一个排序函数：

```typescript
function agentModuleOrder(
  catCode: string,
  style: ContentStyle,
  userProfile: UserProfile
): ModuleKey[] {
  // 优先级：
  // ① 用户这次明确指定的顺序（对话中说"先写price再写taste"）
```

### 6.2 阶段 1：PRD v2（第 1-2 周）— 保留三面板，建学习骨架

**做什么**：F1~F11 全部（详见 PRD v2）

**关键变化**：
- 三版各自独立排序（F3）——验证 V1 学习 + V2/V3 语料参考的正确性
- 采纳快照 + 导出学习（F4）——打通学习数据流

### 7.1 PRD v2 任务在终局视角下的评估

所有 PRD v2 任务在终局架构下仍然必要——它们是 Agent 的底层能力基础：

| PRD v2 任务 | 阶段 1 | 终局下的复用 |
|------|:--:|------|
| F1 左面板去排序化 | ✅ 照做 | 阶段 3 面板消失，排序逻辑保留在 `agentModuleOrder()` 里 |
| F2 物流售后 prompt | ✅ 照做 | 不变——单版本下 aftercare 仍然是系统自动拼接 |
| F3 三版独立排序 | ✅ 照做（保持三版）| 阶段 3 合并为单版排序。阶段 1 三版并存积累风格 Elo 信号 |
| F4 采纳快照 + 导出学习 | ✅ 照做 | 对话模式下学习信号更丰富，快照机制核心逻辑不变 |
| F5 用户画像类目隔离 | ✅ 照做 | 不变 |
| F6 AI 提炼卖点 | ✅ 照做 | 阶段 3 变成 Agent 的一个工具调用 |
| F7 多规格动态表单 | ✅ 照做 | 阶段 3 表单消失，数据结构保留为 Agent 对话中提取多规格 |

### 7.2 结论

**PRD v2 的 11 项任务全部照做，无需调整。** 它们构建的学习骨架（排序引擎、Elo 评分、信号采集、语料搜索）正是对话式 Agent 自主决策的数据基础。没有阶段 1 的信号积累，阶段 3 的 Agent 就是空壳——问什么都只能回"你要什么风格？我还不了解你。"

## 八、已确认的关键决策

以下决策经 2026-07-20 讨论确认，作为后续设计的约束条件：

| # | 决策 | 结论 |
|:--:|------|------|
| 1 | 阶段 1 保留几个版本 | **三版不变。** 多风格竞争是快速积累 Elo 信号的阶段，等阶段 3 再收拢为单版 |
| 2 | 精编/粗编模式 | **精编 = 按模块组确认（5 组）。粗编 = 仅关键模块停下（按品类可配置，见 COARSE_KEY_MODULES）** |
| 3 | 手动编辑能力 | **保留为可展开抽屉。** 收起时对话为主，运营想手动改时展开 contentEditable |
| 4 | 阶段 3 启动时机 | **等飞轮跑通后。** 五条硬条件全部满足（见 §6.4）：Elo ≥ 30 条 + ≥ 3 类目 + 编辑率连续 5 次 < 30% + 语料 ≥ 20 篇 + V1 被接受 |
| 5 | 模式切换 vs 跳过确认 | **两个独立概念。** 模式切换=全局默认变更（工具栏手动操作）；跳过确认=仅当前笔记一次生效（对话中说"剩下的直接写"）|
| 6 | 单版本 Elo 适配 | **导出频次模拟对局。** Elo 算法不变，信号来源从成对比较改为导出频次对比 |
| 7 | 三面板回退 | 阶段 3 保留 `[切回三面板]` 入口，稳定 30 天后移除 |
