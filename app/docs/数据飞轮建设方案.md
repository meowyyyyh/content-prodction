## 一、总览：三层飞轮架构

```
运营使用工具（三版同时生成）
  V1 = 默认风格（固定位置，动态进化，越来越像"你"）
  V2 = Elo 排名第 1 的纯风格
  V3 = Elo 排名第 2 的纯风格
         │
         ▼
┌─────────────────────────────────────────────┐
│              信号采集层                       │
│  显式信号（用户主动表态：采纳、点踩、导出）        │
│  隐式信号（系统自动采集：编辑距离、快捷按钮、重生成） │
│  结果信号（终稿 = 终极验证）                     │
└─────────────────────────────────────────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌──────────┐ ┌─────────────────────────────────┐
│ V1 进化   │ │       信号 → 改进 三层飞轮        │
│          │ │                                  │
│ 模块偏好  │ │ L1 秒级 采纳/点踩/改写 → Elo 调权  │
│ 来源跟踪  │ │        → V2/V3 动态排序            │
│          │ │        → V1 模块偏好累积            │
│ L2 画像  │ │        存储：localStorage           │
│ 参数注入  │ │        范围：个人适配               │
│          │ │                                  │
│ → V1 越来越│ │ L2 分钟级 编辑diff → LLM画像分析    │
│  像"你"   │ │        存储：服务端                 │
└──────────┘ │        范围：个人适配               │
             │                                  │
             │ L3 天/周级 终稿→预审→RAG入库        │
             │        存储：corpus_index.json     │
             │        范围：全局优化               │
             └─────────────────────────────────┘
         │
         ▼
  生成质量提升 → 运营更愿意用 → 用得更多 → 信号更多
```

**三版定位**：

| 版本 | 位置 | 内容 | 变化机制 |
|------|:--:|------|------|
| V1 默认风格 | 固定第 1 | 初始小红书，随时间进化为个人复合风格 | 模块偏好 + L2 画像 → 每个模块独立学习来源风格 |
| V2 | Elo 第 1 | 纯风格（闺蜜风/高端风/…） | Elo 排名决定谁在这个位置 |
| V3 | Elo 第 2 | 纯风格 | 同上 |

**核心原则**：

- **V1 = 个人进化**：每个模块从六种纯风格中学习，加 L2 画像参数，变成"这个人的风格"
- **V2/V3 = Elo 竞争**：六种纯风格按 Elo 排序，排名高的进入默认展示位
- **L1 + L2 = 个人适配**：A 的 V1 和 B 的 V1 完全不同，不互相影响
- **L3 = 全局优化**：大家一起把语料池做大，品类知识全体共享
- **中栏是所有修改的汇聚点**：手动打字、快捷按钮、对话框——三条路最终都走到 `onEdit`，编辑距离统一计算

## 二、完整行为链路

逐条行为到改进结果的完整映射：

| # | 用户行为 | 产生什么数据 | 怎么改进系统 | 产品变好（用户感知） | 怎么带来更多行为 |
|:--:|------|------|------|------|------|
| 1 | 点「采纳此版本」 | {品类, 风格, 模块列表} | 该品类该风格 Elo 加分 | 下次同品类默认用被采纳的风格 | 不用手动调风格 |
| 2 | 点「采纳」某模块 | {品类, 模块, 版本来源风格} | 不影响 Elo（模块采纳是编辑行为） | — | — |
| 3 | 👎 点踩版本 | {品类, 风格} | 该品类该风格 Elo 扣分（惩罚 > 采纳奖励） | 不合适的风格不再出现 | 减少失望 |
| 4 | 👎 点踩模块 | {品类, 模块, 版本来源风格} | 该风格 Elo 扣分（力度同点踩版本） | 这个风格排名下降 | 该风格不会再出现在默认三版 |
| 5 | 中栏快捷按钮 | {按钮类型, 当前品类, 当前风格} | 统计运营最常优化什么（扩充/口语化/排版/…） | 按需求调默认参数 | 少点按钮，一次到位 |
| 6 | 手动编辑模块 | {模块, 原文, 改后文, diff%} | **编辑率 = 质量真分数**；LLM 异步分析提取写作偏好 | 下次该模块更接近运营风格 | 改得少 = 效率高 |
| 7 | 中栏删除模块 | {品类, 模块} | 记录删除事件（PM 参考，不做自动干预） | — | — |
| 8 | 1分钟内重新生成 | {品类, 上一轮三版文案, 参数} | 三版 Elo 各扣分（整体不满意） | 触发阈值后自动换策略 | 不失望走人 |
| 9 | 导出 | 终稿全文 + 元信息 | → 预审库 → 审核通过 → RAG +1 | 同品类检索命中率提升 | 质量高了更愿意导出 |
| 10 | 去发布（未来） | 终稿 + 快团团实际转化 | 知道"真正好用的文案"长什么样 | 生成策略向高转化靠拢 | 正向循环 |

| 模块删除 | 中栏删模块 | 行为 | ✅ 有操作但没记录 | 记录删除事件 |

### E. 对话框自由输入（管"chip 覆盖不到的盲区"）

| 锚点 | 触发条件 | 信号类型 | 当前状态 | 待办 |
|------|------|:--:|:--:|------|
| 自由输入 | 非 chip 触发的对话框发送 | 隐式 ❌ | ❌ 没做 | 记录原文 + 目标模块 + 时间戳 |

自由输入是发现 prompt 盲区和缺失 chip 的重要信号源。采集方案：

```
发送时写入 localStorage['flywheel_chat_log:{userId}']：
  { text, module, time }

导出时随 POST /api/signals/report 一并上报到服务端
  → data/signals/{userId}/chat_log.jsonl（append-only）

看板展示时：
  → 原文截断列表 + LLM 聚类摘要（按需求主题分组）
  → 自由输入占比 > 30% 时，触发聚类分析 → PM 决定加 chip 或调 prompt
```

## 四、权重系统设计（核心机制）

### 设计原则

| 原则 | 说明 |
|------|------|
| 单一 Elo 评分 | 版本级和模块级的采纳/点踩统一汇入风格 Elo（三版天然竞争关系），模块不设独立评分 |
| 正负信号不对称 | 点踩惩罚 > 采纳奖励（不满意比满意更有信息量） |
| 中栏编辑距离是真相 | 右侧采纳/点踩只是态度表态，编辑率才是真正的质量分数 |
| 时间衰减 | 旧信号打折扣（prompt 一直在改进，7 天前的评价可能不适用于当前版本） |
| 冷启动平滑 | 没数据的组合从中间值起步，不做极端判断 |
| 信号可撤回 | 用户误操作后 30 秒内允许撤销，被撤回的信号不计入评分 |
| 最小样本门槛 | 某品类×风格组合信号数 < 3 时，回退到默认值，不读取评分 |

### 版本级 Elo 评分规则

操作 → Elo 变化：

| 事件 | Elo 计算 | 说明 |
|------|------|------|
| 采纳版本（V2/V3） | 该版本战胜另一个版本 | V2/V3 竞争，被选中的胜出；采纳 V1 不计入 Elo |
| 快捷按钮改写为其他风格 + 导出 | **目标风格**战胜当前排位对手，权重 ×2 | 运营主动变换风格，意愿比初始采纳更强 |
| 点踩版本（V2/V3） | 该版本输给中性基准（Elo=1500 的虚拟对手） | 明确否定；点踩 V1 不计入 Elo |
| V2/V3 都没采纳 + 1min 内重生成 | V2/V3 都输给中性基准 | 整体不满意 ⚠️ 含误触保护：距上次生成 < 15s 视为手滑，不扣分 |
| 导出 | 终稿风格额外加分 | 最强正信号；导出时追溯 → 如果最终风格 ≠ 初始采纳风格，改写目标风格额外 +50 Elo |

**例子：快捷按钮改写驱动的 Elo 排名替换**（以高端风为例，同样适用于全部 6 种风格）：

```
运营 A，调制乳品类，初始：V1=默认风格(小红书)，V2=闺蜜风，V3=趣味风

第 1 次：采纳 V2（闺蜜风）→ 点「高端风」改写 → 导出
  → 高端风 +4（改写 +2, 导出 +2）
  Elo: 闺蜜 1500, 趣味 1500, 高端风 1504, 简约 1500, 团长 1500

第 3 次（重复模式）：
  Elo: 闺蜜 1500, 高端风 1512, 趣味 1500

  高端风 Elo 超过趣味 → 排名第 2
  → V3 变成高端风，V2=闺蜜风不变

第 5 次：
  Elo: 高端风 1520, 闺蜜 1500

  高端风 Elo 超过闺蜜 → 排名第 1
  → V2=高端风，V3=闺蜜风

V1 默认风格始终不动——但它会从这些操作中学习：运营总采纳高端风的 price、闺蜜风的 taste。
V1 的 price 模块来源逐步切换为 premium，taste 来源逐步切换为 girlfriend。
```

**关键规则**：
- 全部 6 种风格平等参与 Elo 排名，默认三版取 top 3，其余 3 种通过快捷按钮改写累积分
- 改写只在导出时才入 Elo，中间试来试去不导出 = 不计分
- 只追溯"初始采纳了什么"和"导出时最终风格"，中间变换过程忽略
- 改写目标风格拿 2 倍权重，因为运营为它多花了一步操作——这是比被动接受三选一更强的偏好信号

**K 因子自适应**：

| 该品类×风格数据量 | K 值 | 阶段 | 效果 |
|:--:|:--:|------|------|
| < 5 | 48 | 探索期 | 快速收敛到真实水平 |
| 5 ~ 20 | 24 | 成长期 | 兼顾收敛速度与稳定性 |
| > 20 | 12 | 稳定期 | 微调，不因单次操作跳变 |

### 模块级信号

模块级点踩/采纳不单独维护评分体系，而是作为风格 Elo 的增强信号：

| 事件 | Elo 影响 | 说明 |
|------|------|------|
| 采纳某模块 | 不影响 Elo | 模块采纳是编辑行为，不代表对风格的偏好 |
| 点踩某模块 | 该模块所属风格 Elo -25（≈ 版本点踩力度） | "这个风格的这个模块写得太差了" → 否定的是这个风格 |
| 删除中栏模块 | 不影响 Elo | 可能是模块本身不需要，不归因于风格 |

模块级别的点踩不做自动缩字、跳过等干预——仅影响 Elo 排名。生成的文字如何变化、怎么写出特点，是 PM 看到看板数据后由人决定改 prompt，不由系统自动改。

**中栏编辑距离系数**（影响采纳版本的 Elo 权重）：

⚠️ 关键时序设计：采纳时快照原文，导出时对比快照 vs 定稿

运营的编辑行为可能跨多轮——采纳后在中栏反复改了一下午才导出。如果拿"导出时刻的最新 AI 原文"来算编辑率，会得到虚假的高编辑率。

正确做法：运营点"采纳"时立即快照当前 AI 原文，存到编辑事件的 payload 里，导出时拿定稿 vs 快照算编辑率。

```
编辑率 = (定稿字数 - 快照字数) / 快照字数
```

| 编辑率 | 系数 | 含义 |
|:--:|:--:|------|
| < 20% | ×1.5 | AI 写得好，几乎没改 → 信号放大 |
| 20% ~ 50% | ×1.0 | 正常修改量 → 信号照常 |
| 50% ~ 80% | ×0.3 | 改了大半 → 信号大幅打折 |
| > 80% | ×(-0.5) | 几乎重写 → 采纳变惩罚，运营只是拿它当草稿 |

**关键洞察**：采纳了但改了一大半 ≠ 满意。编辑距离过滤"假采纳"，但前提是快照取自采纳时刻，而非导出时刻。

### 时间衰减

```js
function decayedScore(entry) {
  const days = (Date.now() - entry.last_updated) / 86400000
  const decay = Math.max(0.3, 1 - days * 0.10)  // 每天衰减 10%，最低剩 30%
  return entry.score * decay
}
```

**为什么需要时间衰减？** Prompt 在持续改进。7 天前的点踩可能是因为当时的 prompt 有 bug，不能让它永远拉低分数。如果持续被采纳 → 分数被不断刷新 → 不衰减。

**但时间衰减单独不够。** 旧 prompt 的点踩 → Elo 偏低 → 风格排不进 top 3 → 运营永远看不到优化后的版本 → 永远不会重新采纳 → 旧信号虽然衰减到 30% 但永远不会被刷新。这是一个死循环。

解法分两层：

**① prompt 版本戳**（自动，Phase 1 就做）：

| 机制 | 说明 |
|------|------|
| 每条点踩记录带 `promptVersion` | 信号写入时自动记下当前 prompt 版本号 |
| 提示词更新后版本号递增 | 研发调完 prompt → `style_prompt_version.xiaohongshu += 1` |
| 旧版本信号加速衰减 | 信号来自旧版本 → 衰减斜率从 10%/天 提升到 50%/天（最低 0，2 天归零） |
| 跨大版本直接归零 | prompt 版本跨度 ≥ 2（如 v1 → v3）→ 该风格的旧信号全部作废 |

```js
function decayedScoreV2(entry, currentVersion) {
  const days = (Date.now() - entry.last_updated) / 86400000
  const versionGap = currentVersion - (entry.promptVersion || 1)

  if (versionGap >= 2) return 0                              // 跨大版本 → 归零
  if (versionGap === 1) {
    const decay = Math.max(0, 1 - days * 0.50)               // 旧版本 → 50%/天，2 天归零
    return entry.score * decay
  }
  const decay = Math.max(0.3, 1 - days * 0.10)              // 同版本 → 正常衰减
  return entry.score * decay
}
```

**② 探索机制**（Phase 1 同时做）：

| 机制 | 说明 |
|------|------|
| 10% 探索概率 | 每次一键生成时，10% 概率将第三版换成一个 ELO 偏低的风格 |
| 探索到就计分 | 如果探索版被采纳 → 新信号刷新旧分，死循环自然打破 |
| 运营无感 | 三版仍然三版，只是偶尔换一个进来"试试"，运营不被告知 |

**效果**：prompt 优化后，旧版本的点踩在 2 天内归零（50%/天），加上 10% 探索概率让优化后的风格有机会被重新看到。一旦被采纳，新信号直接覆盖旧分。

### 编辑快照数据结构

编辑率计算依赖"采纳时原文"和"导出时定稿"的对比。因此需要在采纳时存一份快照：

```js
// localStorage['flywheel_snapshots:{userId}']
{
  "active": {
    "调制乳/风味牛奶_2026-07-15_1430": {
      "createdAt": 1721000000000,
      "style": "girlfriend",
      "modules": {
        "taste": { "snapshot": "...采纳时的原文...", "adoptedAt": 1721000000000, "sourceVersion": 2 },
        "price": { "snapshot": "...", "adoptedAt": 1721000005000, "sourceVersion": 2 }
      }
    }
  }
}
```

```
流程：
  1. 运营点「采纳」→ 原文写入 snapshots.active[品类_时间].modules[模块]
  2. 运营编辑中栏 → 随时修改，不影响快照
  3. 运营点「导出」→ 取快照原文 vs 中栏定稿，计算编辑率
  4. 编辑率写入该次采纳的信号记录，快照从 active 中清除
```

### 完整评分公式

```
风格 Elo 变化 = 

  采纳版本        → +K（战胜另外两版各一场）
  点踩版本/模块   → -K（输给中性基准）
  快捷按钮改写+导出 → 目标风格 +2K（改写意愿 > 被动接受）
  导出（终稿风格） → +50（最强正信号）
  三版都没采纳+重生成 → 三版各 -K（整体不满意）

  K 值根据该风格的累计信号数自适应（见 K 因子表）
```

模块不设独立评分体系。模块点踩 = 对该风格的否定，走 Elo 通道，力度与版本点踩一致。模块采纳 = 纯编辑行为，不计入 Elo。

### 权重如何影响 AI 生成

**版本级**（Elo 排名，仅影响 V2/V3）：

```
V1（默认风格）固定不动，不参与 Elo 竞争。
V2 = Elo 排名第 1 的纯风格
V3 = Elo 排名第 2 的纯风格

全部 6 种风格平等参与 Elo 排名，包括通过快捷按钮改写触发的风格。
下次一键生成时：
  → 读取 style_elos → 按 Elo 排名取 top 2
  → 分别作为 V2 / V3
  → 某个风格 Elo 持续 < 1300 → 从候选池移除
```

**模块级**（不自动干预生成）：

模块点踩只影响风格 Elo——同一个模块被反复点踩说明这个风格有问题，风格排名自然会下降。但**不会自动缩短、跳过或取消勾选该模块**。怎么写好这个模块，是 PM 看数据后手动调 prompt 的事。

**快捷按钮统计**（需求推断）：

```
「📝 扩充」占比最高且某模块编辑率高 → 该模块默认字数偏少
  → buildPrompt 时提高该模块最低字数
  → MODULE_CONTENT_REQS 中 "不低于120字" → "不低于200字"

「💬 口语化」占比高 → 部分风格书面语太重
  → 调整对应风格的 prompt 增加口语化程度
```

### 信号撤回（Undo）机制

用户误点采纳/点踩后 30 秒内允许撤销，被撤回的信号不计入评分：

| 步骤 | 行为 |
|:--:|------|
| 1 | 用户操作 → 正常写入 localStorage，同时前值 JSON 快照存入 `flywheel_undo:{userId}` |
| 2 | 30 秒内点"撤销" → 从 undo buffer 恢复前值，丢弃刚写入的记录 |
| 3 | 30 秒后或下一次新操作 → undo buffer 自动覆盖 |

```js
// undo buffer 数据格式
localStorage['flywheel_undo:default'] = {
  timestamp: 1721000000000,
  expireAt: 1721000030000,          // 写入时刻 + 30s
  previousStyleElos: { ... },
  previousModuleScores: { ... },
  rolledBackSignal: { type, details }
}
```

**UI 交互**：操作后按钮附近出现微型 toast "已记录 · 撤销"（3 秒自动消失）。**幂等性**：撤销直接写回前值快照，不重新触发 Elo 计算，同一条信号不可撤销两次。

### 默认风格：第七种风格，动态进化

默认风格不是六种纯风格之一。它是每个运营专属的第七种——初始模板用小红书种草风，然后从运营的每一次操作中学习进化。

**和六种纯风格的区别**：

| | 六种纯风格 | 默认风格 |
|------|------|------|
| 本质 | 静态 prompt 模板 | 动态 prompt 生成器 |
| prompt | 研发写死在后端 | 每次生成前根据用户行为数据实时拼装 |
| 粒度 | 全篇统一风格 | 每个模块可来自不同风格 |
| 变化 | 研发手动改才变 | 运营每操作一次，下次就可能变 |
| 共享 | 所有用户同一套 | 每人一套，互不相同 |

**进化机制**：

```
运营从 V2（闺蜜风）采纳了 taste 模块 → 重复多次
  → 默认风格的 taste 模块来源 = "girlfriend"
  → 下次生成 V1 时，taste 用闺蜜风的规则写

运营从 V3（高端风）采纳了 price 模块 → 重复多次
  → 默认风格的 price 模块来源 = "premium"
  → 下次生成 V1 时，price 用高端风的规则写

运营在中栏总是把 hook 的 emoji 删掉、把"姐妹们"改掉
  → L2 画像提取：emoji 低、不用"姐妹们"
  → 注入 V1 prompt，叠加到所有模块
```

**数据结构**：

```js
// localStorage['flywheel_scores:{userId}'] 新增
"default_modules": {
  "食品保健/咖啡麦片冲饮/常温乳制品/调制乳-风味牛奶": {
    "taste":  { source: "girlfriend",   n: 12 },
    "price":  { source: "premium",      n: 8  },
    "hook":   { source: "xiaohongshu",  n: 15 },
    "trust":  { source: null,           n: 0  },   // 无偏好，回退基础风格
    // ... 14 个模块
  }
}
```

**冷启动与退化**：

| 场景 | 行为 |
|------|------|
| 新用户 / 该模块从未被采纳 | 使用初始基础风格（小红书种草风） |
| 该模块被采纳 ≥ 5 次且 ≥ 80% 来自同一风格 | 完全使用该风格的规则 |
| 被采纳 3-4 次，来源分散 | 使用偏好风格规则，语气向基础风格靠拢 20% |
| 被采纳 < 3 次 | 不回退，仍用基础风格 |

**prompt 拼装**（每次生成 V1 时实时计算）：

```
System: 生成以下模块。这是为这位运营定制的默认风格。

  → hook（来源：小红书种草风，n=15）：
    按小红书种草风 hook 规则，但叠加 L2 画像参数：
    - 不用"姐妹们"开头
    - emoji 密度从高降为中
    - 段落 2-3 句

  → taste（来源：闺蜜风，n=12）：
    按闺蜜风 taste 规则，叠加 L2 画像参数：
    - 口语化，"我跟你说"开头
    - 扩充到 250 字

  → price（来源：高端风，n=8）：
    按高端风 price 规则：
    - 直接报价格规格，不渲染
    - L2 画像无冲突参数，全部注入
```

**和 Elo 的关系**：两条线独立运作。

```
默认风格 → 模块级偏好（每个模块独立跟踪来源风格）→ V1 进化
Elo 排名 → 版本级采纳/点踩/改写 → V2/V3 排序

运营从 V2 采纳 taste → 默认风格 taste 偏好 +1 到闺蜜风
运营点踩 V3 → V3 的纯风格 Elo -25 → V3 可能被换掉
```

**Agent 阶段**：默认风格足够成熟时（累计导出 ≥ 30 篇且编辑率 < 20%），六个纯风格可以仅保留为快捷按钮选项，RAG 语料库继续在后台运行。用户看到的只有一个不断进化的默认生成结果。

## 五、三层飞轮：各自怎么转

| 层级 | 触发 | 周期 | 存储 | 范围 | 成本 | 做什么 |
|:--:|------|:--:|------|:--:|:--:|------|
| L1 | 采纳/点踩 | 秒级 | localStorage | 个人 | 零 | Elo 即时调权 → 下次生成版本排序变化 |
| L2 | 导出 | 分钟级 | 服务端用户档案 | 个人 | ~500 tokens/次 | LLM 分析编辑 diff → 提取用户画像 → 下次注入偏好 |
| L3 | 导出+审核 | 天/周级 | corpus_index.json | 全局 | 零 | 终稿入库 → RAG 检索命中率提升 |

### L1 — 秒级：采纳/点踩 → 即时参数调整

```
触发：运营在右侧面板点"采纳版本" / "点踩版本" / "点踩模块" / "采纳模块"
写入：localStorage['flywheel_scores:{userId}']
生效：下一次点击"一键生成"时读取
影响：
  - V2/V3 风格由 Elo 排名动态决定（6 种纯风格 top 2）
  - V1 默认风格各模块来源由模块级采纳偏好决定
  - L2 用户画像参数注入 V1 的所有模块
周期：秒级（前端 localStorage，无需后端）
范围：个人（A 的采纳不影响 B）
成本：零
```

### L2 — 分钟级：编辑行为 / 快捷按钮 → 用户画像

```
触发：运营导出时
采集：{原文, 定稿, diff%, 按钮使用频率}
分析：后台异步用 DeepSeek 分析编辑 diff
      → 提取结构化偏好参数：
        {
          段落偏好: "短段落",
          emoji 密度: "低",
          口癖关键词: ["我跟你说"],
          高频按钮: ["口语化", "去 emoji"],
        }
注入：下次生成时 System Prompt 追加偏好参数
      "你正在为一位偏好口语化短句、不喜欢感叹号的运营写作"
周期：分钟级（导出时异步分析，下次会话生效）
范围：个人（A 的画像只影响 A 的生成）
成本：每次分析约 500 tokens（DeepSeek 极便宜）

风格 > 用户画像（冲突仲裁规则）：
  核心原则：风格定骨骼，画像填血肉。
  用户画像不能覆盖风格的硬性特征，只填充风格未明确指定的参数区间。

  仲裁流程：
    1. 风格模板自带一份"显式声明参数表"——哪些参数是该风格的身份特征（emoji 频率、语气基调、句式结构等）
    2. 用户画像参数与声明表做交集检查
    3. 冲突参数 → 跳过，不注入
    4. 无冲突参数 → 注入 System Prompt

  示例仲裁结果：

  | 参数 | 小红书风声明 | 用户画像 | 结果 |
  |------|:--:|:--:|:--:|
  | emoji 密度 | 高频（身份特征） | 低 | 风格优先，跳过 ❌ |
  | 段落长度 | 未声明 | 短段落 | 注入 ✅ |
  | 语气夸张度 | 高（身份特征） | 温和 | 风格优先，跳过 ❌ |
  | 口癖关键词 | 未声明 | ["我跟你们说"] | 注入 ✅ |
  | 开头句式 | 未声明 | 问句开头 | 注入 ✅ |

  如果全部参数都冲突 → 说明运营本质上是"不该用这个风格的人"，不动用户画像，让 L1 的 Elo 排名把更合适的风格排到前面
```

### L3 — 天/周级：导出终稿 → 预审 → 入库 → RAG

```
触发：运营导出终稿
写入：data/review_queue/YYYY-MM-DD-商品名.json
审核：人工在预审库里审核
  → ✅ 通过：运行 import_corpus.cjs → corpus_index.json 更新
  → ❌ 不通过：移到 rejected/ 或删除
注入：下次生成同品类时，RAG 检索命中率提升
周期：天级（审核）+ 周级（同品类 ≥ 5 篇语料 → 检索质变）
范围：全局（所有运营共享语料池，但只共享"好文案长什么样"，不共享个人偏好）
成本：几乎为零
```

## 六、个人适配 vs 全局优化

### 核心分界

| 层级 | 个人还是全局 | 存储在哪 | 影响范围 | 例子 |
|------|:--:|------|------|------|
| L1 权重 | **个人** | localStorage（按 userId 隔离） | 只影响这一个运营 | A 的采纳让闺蜜风排第一，B 不受影响 |
| L2 画像 | **个人** | 服务端用户档案 | 只影响这一个运营 | A 不写"姐妹们"，B 照常用 |
| L3 语料 | **全局** | corpus_index.json | 所有人共享 | A 导出的优质闺蜜风稿，B 生成时也可参考 |

### 为什么要隔离

假设两个运营偏好完全相反：

```
运营 A：喜欢闺蜜风、taste 短文案、不用 emoji
运营 B：喜欢小红书风、taste 长文案、emoji 拉满

如果混在一起求平均：
  → 风格 = 不知道该用哪个
  → taste 字数 = 不长不短（两人都不满意）
  → emoji = 互相抵消，白干
```

**个人权重 = 系统适应每个运营，而不是求一个"平均运营"。**

### 实现

```js
	// 浏览器端（localStorage，个人隔离，运行时读取）
	const userId = 'default'  // 现在匿名模式，多租户后从登录态获取
	⚠️ 当前"匿名模式"下的 localStorage 是伪隔离：所有运营共用同一浏览器/profile，
	`flywheel_scores:default` 其实是多人在同一 localStorage 上混写。
	在正式上线多租户登录之前，真实隔离不可达。
	建议短期规避方案：
	  1. 在 Storage key 中追加设备标识（如 `flywheel_scores:laptop-meowyh`）→ 按设备隔离
	  2. 每个运营在工具开始前输入一个简短代号（如 "A" / "B"）→ 写入 `flywheel_scores:A`
	两者都无需服务端改造，只需前端加一行输入
const scores = JSON.parse(
  localStorage.getItem(`flywheel_scores:${userId}`) || '{}'
)

// 服务端（上报汇总，PM 看板数据源）
// 每次导出时静默上报 → POST /api/signals/report
//   → 写入 data/signals/{userId}.json
//   → PM 打开看板时 GET /api/signals/dashboard → 聚合所有用户
```

### 语料库为什么是全局的

语料库存的是"好文案长什么样"，不存"谁写的、喜欢什么风格"。检索时按品类匹配，不按人匹配。A 导出的终稿 = 这个品类下的好文案范例，B 生成同类目时可以参考——**但不是把 A 的偏好强加给 B**。

### 类目扩展与信号继承

当前四级叶子类目只有一个（调制乳/风味牛奶），后续扩展到一级类目（食品保健）后需要考虑两个维度的差异：

| 维度 | 跟类目走？ | 原因 |
|------|:--:|------|
| 风格 Elo 排名 | ❌ 跨类目共享 | 风格偏好是人的写作习惯——乳品喜欢闺蜜风的人，写零食大概率也用闺蜜风 |
| 模块事件计数 | ✅ 按类目隔离 | 同是 taste 模块，酸奶的口感描述和坚果完全不是一回事 |

```
类目层级示例：
  食品保健（一级）
    ├─ 咖啡/麦片/冲饮（二级）
    │   └─ 常温乳制品（三级）
    │       ├─ 调制乳/风味牛奶（四级）← 当前唯一
    │       └─ 纯牛奶（四级）         ← 未来新增
    └─ 休闲零食（二级）
        └─ ...
```

**风格 Elo**：全类目共享一份，运营在乳品里养成的偏好直接带入零食，不需要重新教。

**模块事件计数**：按叶子类目隔离，新类目冷启动时向上继承父级：

| 场景 | 行为 |
|------|------|
| 纯牛奶首次生成 | 模块事件计数读父级"常温乳制品" |
| 常温乳制品也无数据 | 继续上溯 |
| 全链无数据 | 使用默认模块权重 |
| 纯牛奶积累 ≥ 10 次信号 | 完全独立，不再继承 |

```
自身信号权重 = min(n / 10, 1)  // 0 → 权重 0，完全继承；10+ → 权重 1，完全独立
```

**数据结构**：风格 Elo 扁平结构（key = 风格名），模块事件计数按类目路径存储，读时按前缀向上匹配。

**看板影响**：看板一（模块健康度）和看板二（风格适配度）加"类目层级"下拉框，PM 可选择看某个叶子类目或向上汇总。

## 七、PM 数据看板（产品经理专用）

### 为什么不需要神策等第三方平台

神策擅长的是通用行为分析（PV/UV、页面停留、点击漏斗、用户路径），但我们的数据结构是自定义的（Elo 等级分、模块采纳/点踩计数、编辑率、个人画像参数），接入神策等于把自定义数据硬塞进通用框架——不匹配，也用不上。

**真正需要的就两样**：浏览器端 localStorage（个人 L1 数据） + 服务端 JSON 文件（聚合多运营数据）。不需要数据库、不需要第三方 SDK。

┌──────────────────────────────────────────────────┐
│          服务端 data/signals/                      │
│  default.json  ← 运营 A 的信号数据                  │
│  user-b.json   ← 运营 B 的信号数据（多租户后）       │

### 上报端点（Phase 0 即可加）

```js
// POST /api/signals/report — 每次导出时浏览器自动调用
app.post('/api/signals/report', (req, res) => {
  const { userId, signals } = req.body
  // signals = { style_elos, module_events, button_usage, export_history }
  
  // append-only 写入：每一条信号追加到 log 文件，不做原地合并
  // 避免多运营同时上报时的读写竞争（读取-合并-覆盖 → 后写入者可能丢失前者的数据）
  writeSignalLogFile(userId, signals)
  
  res.json({ success: true })
})

// GET /api/signals/dashboard?days=7 — PM 看板数据
// 读时聚合：读取所有 log 文件，实时聚合计算，不维护增量缓存
// 好处：① 无数据竞争 ② 天然支持时间回溯（可算任意时间窗口）
app.get('/api/signals/dashboard', (req, res) => {
  const days = parseInt(req.query.days) || 7
  const cutoff = Date.now() - days * 86400000
  const allUsers = readAllSignalLogFiles(cutoff)  // 只读指定时间窗口内的 log 条目
  const dashboard = aggregateDashboard(allUsers, days)
  res.json(dashboard)
})
```

上报时机：导出 .doc 时顺便 `fetch('/api/signals/report', ...)`，异步、静默、不阻塞导出。

**append-only 模式说明**：

```
data/signals/
├── default/              # 每个用户一个独立目录
│   ├── 2026-07-14.jsonl  # 每天一个 JSONL 文件（每行一条信号记录）
│   └── 2026-07-15.jsonl
└── user-b/
    ├── 2026-07-14.jsonl
    └── 2026-07-15.jsonl
```

vs 原来的 read-merge-write：
- 单文件读写 → 并发覆盖（多人同时上报时后写覆盖前写）
- append-only → 每条信号追加到文件末尾，不存在覆盖
- 读时聚合 → 取回所有 log 条目，实时计算汇总值
- 额外好处：可以回溯任意时间窗口的历史状态，看板天然支持历史对比

### PM 如何使用

```
打开看板页面（独立 URL 或中栏隐藏入口 + ?pm=1）
  → 默认显示最近 7 天
  → 可自由切换时间粒度：
      按天（最近 1/3/7 天）— 适合排查"昨天改了 prompt，今天效果怎么样"
      按周（最近 1/2/4 周）— 日常复盘最常用
      按月（最近 1/3 月）  — 看长期趋势
      自定义起止日期        — 任意时间段
  → 可筛选品类
  → 四个看板一目了然
  → 发现问题 → 调 prompt → 下周看数字有没有变绿
```

| 粒度 | 适用场景 | 注意 |
|------|------|------|
| 按天 | 调完 prompt 次日快速验证 | 单运营每天 3~5 篇，样本少时波动大，看采纳率不如看编辑率 |
| 按周 | 日常复盘（默认） | 最常用的粒度，样本量刚好够统计意义 |
| 按月 | 季度复盘、风格适配度评估 | 看长期趋势，不会因单周波动误判 |

**不需要定时任务**：看板打开时实时聚合计算，根据选择的时间窗口只读对应日期范围的 JSONL 文件。数据量很小（每人每天几 KB），毫秒级算出。

### 技术实现（分阶段）

**Phase 0（MVP 期间）**：加好上报端点 `POST /api/signals/report`，导出时自动上报。服务端存 JSON 文件积累数据。只看不读——信号就这么默默攒着。

**Phase 1-3（飞轮逐步开启）**：上报一直在跑，服务端数据持续累积。此时你可以在 `data/signals/` 目录下看到每个人的 JSON 文件。

## 八、冷启动：极速闭环策略

### 为什么不需要"冷启动"

**团队自己的运营就是种子用户**。不需要去外面找，内部 1-2 个运营每天用，一周就能看到变化。

### 极速闭环节奏

| 时间线 | 研发侧 | 运营侧 | 飞轮效果 |
|:--|------|------|------|
| 每天 | — | 用工具生成 3-5 篇 | 信号积累 |
| 当天 | 运营说"XX 模块不对" → 下午调 prompt | 反馈使用感受 | 即时改进 |
| 每天 | — | 导出终稿 | → 预审库 |
| 每周 | 看板复盘：编辑率降了没？点踩少了没？ | — | 数据驱动决策 |

### 种子语料人工筛选

冷启动期运营需要从历史笔记中人工筛选种子语料，使用《种子语料入库评分体系设计方案》中的 5 维度评分卡打分。入库线：总分 >= 3.5 分。详见 docs/种子语料入库评分体系设计方案.md，配套 Excel 模板见 docs/种子语料入库评分表.xlsx。

### 核心优势

**运营和研发在一个团队，不需要跨部门审批、排期、PRD。上午反馈下午改。这是最大的加速度。**

### 最小样本门槛

某风格第一次被采纳可能是运气。信号不足时开启 Elo 排序会把偶然好分数当成真实水平。L1 仅在以下条件**全部满足**时才读取权重：

| 维度 | 门槛 | 说明 |
|------|:--:|------|
| 风格 Elo | 该用户累计信号数 ≥ 5 | 跨类目汇总，最少 5 次操作才有统计意义 |
| 模块事件计数 | 该品类×模块组合信号数 ≥ 3 | 按叶子类目独立计数 |

不满足时回退到默认值——风格排序 `['xiaohongshu', 'girlfriend', 'fun']`，模块权重默认——权重系统**只写不读**。Phase 1 用硬阈值，数据量上来后可升级为贝叶斯置信区间。

## 九、质量过滤：拒绝垃圾数据

### 入库前的四道滤网

| 滤网 | 规则 | 处理方式 |
|------|------|------|
| 编辑率过滤 | 编辑率 > 70% → 不入库 | AI 写的几乎被重写 = 样本不可用 |
| 导出超时 | 生成完 30 分钟没导出 | 可能只是试试，不算真正使用 |
| 重复过滤 | 与已有语料相似度 > 90% | 同质内容不重复入库 |
| 人工预审 | 在 review_queue 里审核 | 最后一道防线 |

### 简单规则

**导出 + 编辑率 < 50% = 正样本。其他都不入库。** 过滤掉运营随手生成、翻了两页就走的那些。

**语料来源区分**：自动入库的笔记标记 `source: "auto"`，人工经入库评分卡筛选入库的标记 `source: "curated"`（详见《种子语料入库评分体系设计方案》）。两类语料在动态评分系统中起点不同——curated 语料以入库评分卡总分作为初始种子值，冷启动期优先引用。

## 附录：关键术语速查

| 术语 | 含义 |
|------|------|
| 显式信号 | 用户主动表达的态度（采纳、点踩、导出） |
| 隐式信号 | 系统自动采集的行为（编辑率、重生成、快捷按钮） |
| 结果信号 | 终稿 = 被验证过的好文案（导出、发布） |
| Elo 等级分 | 竞争性评分系统，三版天然比拼 |
| 编辑率 | (定稿字数 - 原文字数) / 原文字数，衡量 AI 写得有多接近终稿 |
| 编辑距离系数 | 编辑率 → 信号权重映射（改得越少 = 信号越真实） |
| 时间衰减 | 旧信号每天打 9 折（最低保留 30%），防止过期评价拖累当前 |
| 预审库 | data/review_queue/，导出终稿暂存处，人工审核后入库 |
